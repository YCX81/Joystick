# 工业摇杆产品线软件架构设计

## 一、问题分析

一个工业摇杆产品线通常面临三个维度的变化：

```
维度1：产品型号（硬件差异）
├── JS-100：单轴摇杆，2按键，CAN输出
├── JS-200：双轴摇杆，4按键，CAN输出
├── JS-300：三轴摇杆+旋钮，8按键，CAN+安全功能
└── JS-400：双轴+力反馈，12按键，CAN+安全功能

维度2：通信协议（客户需求差异）— 仅 CAN 总线族
├── CAN 2.0B（自定义协议）
├── CANopen（DS401 / DS406）
├── SAE J1939
└── CANopen Safety（EN 50325-5）

维度3：运行平台（芯片差异）
├── STM32F103
├── STM32G431
├── S32K144
└── 未来平台
```

如果用排列组合，4×4×3 = 48 种配置。传统做法是每种配置一个工程，维护噩梦。
正确的做法是用**组合而非继承**，通过架构设计让这三个维度正交独立。

### 架构演进历程

| 变化点 | v1 设计 | v2 改进 | v3 改进 | v4 改进 | 对应原则 |
|--------|---------|---------|---------|---------|----------|
| 摇杆核心 | 单函数五职责 | 信号处理链 pipeline | 同 v2 | 删除泛型 chain，统一为显式 `axis_pipeline`（`uint16_t→int32_t` 类型分界） | SRP |
| 输入采集 | 直接调 ADC | 同 v1 | `scan_buffer_t` + `board_scanner_t` 扫描-缓冲解耦 | 同 v3 | SRP+OCP |
| 按键采集 | 仅 GPIO | 同 v1 | `buf_btn_source`（MUX 通道阈值检测，支持多键同按） | 同 v3 | ISP |
| 产品配置 | `#elif` 链 | 每产品独立 `.c` | 同 v2（满量程由 `resolution_bits` 推导） | 新增 `protocol_bundle` 字段 + per-axis `max_deflection` | OCP |
| 协议接口 | 7方法全实现 | 能力位掩码 | 同 v2 | 协议引擎只管打包，值域转换移至 `output_adapter` | SRP+Adapter |
| 协议输出 | 值转换嵌入引擎 | 同 v1 | Profile 驱动 | 新增 `output_adapter` 层，协议引擎接收自描述 `output_data_t` | Anti-Corruption |
| ADC 接口 | 胖接口 | ISP 拆分 | 新增 SPI 接口 | 同 v3 | ISP |
| 平台注册 | `#if defined` | `__attribute__((constructor))` | 显式 `platform_init()`（MISRA 合规） | 同 v3 | DIP |
| 归一化范围 | ±10000 固定 | 同 v1 | 内部无符号原始范围，协议层按需转换 | 不归一化；`output_adapter` 在协议打包前一次性缩放 | — |
| 任务架构 | 裸循环 | 裸循环 + `os_delay` | RTOS 多任务（CMSIS-RTOS v2） | safety_task 升级为 L2 独立计算通道 | SRP+安全 |
| 事件总线 | 无 | 同步分发 | 增加 ISR 安全延迟分发 | 同 v3 | 观察者 |
| 协议输出 | CAN + Analog 混合 | 同 v1 | 仅 CAN 族（CANopen/J1939/CAN_Raw） | 同 v3 | 精简 |
| 死区策略 | 硬编码线性死区 | 可插拔策略（线性/S曲线/指数） | 同 v3 | 新增 |
| 功能安全 | 无 | 无 | safety_task 旁观式监控 | E-Gas 三层架构（L1功能/L2独立监控/L3条件喂狗+硬件关断） | IEC 61508 SIL 2 |
| 安全关断 | 无 | 无 | 依赖 event_bus → 状态机 | 独立硬件关断路径（GPIO 直驱安全继电器，不经 RTOS） | PL d Cat.3 |
| 看门狗 | 无 | 无 | 无条件喂狗 | 条件喂狗（L2 比对通过 + 程序流标记正确时才喂） | IEC 61508-7 |

## 三、驱动接口层设计（ISP 改进：按使用场景拆分）

### 3.1 ADC 接口 — 按使用场景隔离

原设计将所有 ADC 功能放在一个 `adc_driver_t` 中。摇杆核心只需 `read()`，安全监控需要 `read_mv()`，DMA 采集才需要 `start_continuous()`。按接口隔离原则拆分为三个独立接口：

```c
// drivers/adc_reader.h — 基础读取接口，大多数模块只需要这个
#pragma once
#include "drv_common.h"

typedef struct {
    drv_status_t (*init)(void);
    drv_status_t (*read)(adc_channel_id_t ch, uint16_t *raw);
    drv_status_t (*deinit)(void);
} adc_reader_t;
```

```c
// drivers/adc_voltage_reader.h — 电压读取接口，安全监控/诊断需要
#pragma once
#include "drv_common.h"

typedef struct {
    drv_status_t (*read_mv)(adc_channel_id_t ch, uint32_t *mv);
} adc_voltage_reader_t;
```

```c
// drivers/adc_continuous.h — DMA 连续采集接口，高速采集模块需要
#pragma once
#include "drv_common.h"

typedef struct {
    drv_status_t (*start_continuous)(adc_channel_id_t *chs, uint8_t cnt,
                                     uint16_t *buf, uint16_t buf_len);
    drv_status_t (*stop_continuous)(void);
} adc_continuous_t;
```

```c
// drivers/drv_common.h — 公共类型定义
#pragma once
#include <stdint.h>

typedef enum {
    DRV_OK = 0,
    DRV_ERROR,
    DRV_TIMEOUT,
    DRV_BUSY,
} drv_status_t;

typedef enum {
    ADC_CH_AXIS_X,
    ADC_CH_AXIS_Y,
    ADC_CH_AXIS_Z,
    ADC_CH_KNOB,
    ADC_CH_VREF,
    ADC_CH_TEMP,
    ADC_CH_SUPPLY,
    ADC_CH_MAX
} adc_channel_id_t;
```

平台实现可以同时满足多个接口：

```c
// platform/stm32g4/adc_stm32g4.c
const adc_reader_t          adc_reader_stm32g4 = { .init = ..., .read = ..., .deinit = ... };
const adc_voltage_reader_t  adc_voltage_stm32g4 = { .read_mv = ... };
const adc_continuous_t      adc_continuous_stm32g4 = { .start_continuous = ..., .stop_continuous = ... };
```

各模块只依赖自己需要的最小接口：

```c
// 摇杆核心只依赖基础读取
void joystick_core_init(joystick_core_t *js,
                         const adc_reader_t *adc,
                         const gpio_reader_t *gpio);

// 安全监控依赖电压读取接口
void safety_monitor_init(safety_monitor_t *sm,
                          const adc_voltage_reader_t *adc,
                          const gpio_reader_t *gpio);
```

### 3.4 平台注册机制（DIP 改进：依赖方向完全反转）

原设计中 `app_config.c` 通过 `#if defined(PLATFORM_STM32G4)` 直接引用平台符号，编译时依赖方向从上往下。改进后，平台层主动向上注册，上层代码完全不知道具体平台的存在。

```
改进前（依赖方向向下）：
  app_config.c ──→ platform/stm32g4/adc_stm32g4.c

改进后（依赖方向向上）：
  app_config.c ──→ drivers/adc_reader.h ←── platform/stm32g4/adc_stm32g4.c
```

```c
// platform/platform_registry.h — 上层只依赖这个抽象接口
#pragma once
#include "adc_reader.h"
#include "adc_voltage_reader.h"
#include "drv_can.h"
#include "drv_gpio.h"
#include "drv_spi.h"

void platform_register_adc(const adc_reader_t *drv);
void platform_register_adc_voltage(const adc_voltage_reader_t *drv);
void platform_register_can(const can_driver_t *drv);
void platform_register_gpio_reader(const gpio_reader_t *drv);
void platform_register_gpio_writer(const gpio_writer_t *drv);
void platform_register_spi(const spi_driver_t *drv);
void platform_register_stl(const safety_stl_t *stl);  // v4: 安全自检库注册

const adc_reader_t*          platform_get_adc(void);
const adc_voltage_reader_t*  platform_get_adc_voltage(void);
const can_driver_t*          platform_get_can(void);
const gpio_reader_t*         platform_get_gpio_reader(void);
const gpio_writer_t*         platform_get_gpio_writer(void);
const spi_driver_t*          platform_get_spi(void);
const safety_stl_t*          platform_get_stl(void);   // v4

// 显式初始化入口 — 每个平台实现一个
// 替代 __attribute__((constructor))，满足 MISRA C Rule 1.2
void platform_init(void);
```

```c
// platform/stm32g4/platform_init.c
// 显式调用，初始化顺序确定性可控，可设断点调试
// 不依赖任何 GCC 扩展，满足功能安全要求

#include "platform_registry.h"
#include "adc_stm32g4.h"
#include "can_stm32g4.h"
#include "gpio_stm32g4.h"
#include "spi_stm32g4.h"

void platform_init(void) {
    platform_register_adc(&adc_reader_stm32g4);
    platform_register_adc_voltage(&adc_voltage_stm32g4);
    platform_register_can(&can_driver_stm32g4);
    platform_register_gpio_reader(&gpio_reader_stm32g4);
    platform_register_gpio_writer(&gpio_writer_stm32g4);
    platform_register_spi(&spi_driver_stm32g4);
    #if SAFETY_ENABLED
    platform_register_stl(&stl_stm32g4);   // v4: 注册 X-CUBE-STL 认证安全库
    #endif
}
```

> **为什么不用 `__attribute__((constructor))`？**
>
> | 对比项 | `__attribute__((constructor))` | 显式 `platform_init()` |
> |--------|-------------------------------|----------------------|
> | MISRA C 合规 | 违反 Rule 1.2（实现定义行为） | 合规 |
> | 初始化顺序 | 不确定，跨编译单元无法保证 | 代码中显式控制 |
> | 可调试性 | 在 `main()` 之前执行，断点难设 | 普通函数调用，随时断点 |
> | 功能安全认证 | IEC 61508 / ISO 13849 审核不通过 | 可审核、可追溯 |
> | 可移植性 | GCC 专属，IAR/ARMCC/Green Hills 不支持 | 纯 C 标准，所有编译器通用 |

```c
// main.c — 启动顺序清晰（使用 CMSIS-RTOS v2 API）
#include "cmsis_os2.h"

int main(void) {
    // 1. 硬件初始化（BSP / CubeMX 生成）
    HAL_Init();
    SystemClock_Config();

    // 2. 平台驱动注册（显式调用，替代 constructor）
    platform_init();

    // 3. RTOS 内核初始化
    osKernelInitialize();

    // 4. 创建应用任务
    app_tasks_create();

    // 5. 启动调度器（不返回）
    osKernelStart();
}
```

链接器决定哪个平台的 `platform_init.c` 参与编译，上层代码零修改。每个平台只需提供自己的 `platform_init()` 实现即可。

## 四、输入采集层（scan_buffer 扫描-缓冲解耦模式）

### 4.1 设计动机

原设计假设"轴 = ADC 通道"，`joystick_update()` 直接调用 `adc->read(ch, &raw)`。但实际工业摇杆的输入源远比这复杂：

```
实际硬件拓扑（以 JS-300 为例）：
         5V ──┬── R1 ──┬── GND     5V ──┬── R1 ──┬── GND
              │ Button │                │ Button │
              2.5V     │                2.5V     │
                ↓      ↓                  ↓      ↓
        ┌──────────────────────────────────────────┐
        │       Analog MUX (CD74HC4067)            │
        │  CH0: Axis-X pot    CH4: Button-0 (2.5V) │
        │  CH1: Axis-Y pot    CH5: Button-1 (2.5V) │
        │  CH2: Axis-Z pot    CH6: Button-2 (2.5V) │
        │  CH3: (reserved)    CH7: Button-3 (2.5V) │
        └──────────────┬───────────────────────────┘
                       │ COM (共用输出)
                       ↓
                  ┌─────────┐
                  │   ADC   │ ← 同一个 ADC 通道
                  └─────────┘
```

关键约束：
1. **轴和按键共用 MUX+ADC 资源**——不能各自独立读取，必须协调扫描顺序
2. **按键采用逐通道 5V→2.5V 分压**——每个按键占独立 MUX 通道，按下时读 ≥2.5V，支持多键同按
3. **不同产品硬件差异大**——有的用 MUX，有的直接 ADC 多通道，有的用 DMA 连续采集

若将 MUX+ADC 作为整体资源封装进输入源，会导致轴源和按键源耦合在同一个扫描器上，缺乏平台化能力。因此需要一个**纯数据解耦点**。

### 4.2 scan_buffer 解耦模式

核心思想：将"谁扫描硬件"和"谁消费数据"通过纯数据结构 `scan_buffer_t` 解耦。

```
                    ┌──────────────────────────────┐
                    │    Board-level Scanner        │
                    │    (board_scanner_t)          │
                    │  ┌──────────┐ ┌───────────┐  │
                    │  │mux_adc   │ │direct_adc │  │   可替换策略
                    │  │_scan     │ │_scan      │  │
                    │  └────┬─────┘ └─────┬─────┘  │
                    │       └──────┬──────┘        │
                    └──────────────┼───────────────┘
                                   │ scanner.scan(ctx, &buf)
                                   ▼
                    ┌──────────────────────────────┐
                    │    scan_buffer_t              │  ← 纯数据，零依赖
                    │    uint16_t values[16]        │
                    │    uint8_t  count             │
                    └──────────┬───────────────────┘
                               │ 只读引用
                    ┌──────────┴───────────┐
                    ↓                      ↓
          ┌──────────────┐       ┌──────────────┐
          │buf_axis_source│       │buf_btn_source │
          │ .read_raw()  │       │ .read_buttons()│
          │ 按 channel   │       │ 阈值检测       │
          │ 索引取值      │       │ 多键同按       │
          └──────┬───────┘       └──────┬────────┘
                 ↓                      ↓
          Signal Chain             Debounce
```

### 4.3 scan_buffer — 纯数据缓冲区

```c
// input/scan_buffer.h — 纯数据结构，零外部依赖
#pragma once
#include <stdint.h>

#define SCAN_BUF_MAX_CHANNELS  16

typedef struct {
    uint16_t values[SCAN_BUF_MAX_CHANNELS];  // 各通道原始 ADC 值
    uint8_t  count;                           // 有效通道数
} scan_buffer_t;
```

`scan_buffer_t` 是整个输入采集层的**解耦核心**：
- 上游（扫描器）只写入它
- 下游（输入源）只读取它
- 它本身不依赖任何硬件接口，可在 PC Mock 上直接填充测试数据

### 4.4 board_scanner — 板级扫描策略

```c
// board/board_scan.h — 扫描策略接口（板级关注点）
#pragma once
#include "scan_buffer.h"

typedef struct {
    void (*scan)(void *ctx, scan_buffer_t *buf);  // 一次扫描所有通道
    void *ctx;                                     // 指向具体策略的上下文
} board_scanner_t;
```

不同产品/硬件使用不同的扫描策略：

```c
// board/mux_adc_scan.c — MUX 选通 + ADC 逐通道采集
#include "board_scan.h"
#include "adc_reader.h"
#include "drv_gpio.h"

typedef struct {
    const adc_reader_t  *adc;          // ADC 驱动接口
    const gpio_writer_t *gpio;         // MUX 选通引脚控制
    adc_channel_id_t     adc_channel;  // MUX COM 对应的 ADC 通道
    gpio_pin_id_t        sel_pins[4];  // MUX 地址选择引脚 (S0-S3)
    uint8_t              sel_pin_count; // 实际使用的选择引脚数
    uint8_t              total_channels;// 需要扫描的总通道数
    uint16_t             settle_us;    // MUX 切换后建立时间 (μs)
} mux_adc_scan_ctx_t;

static void mux_adc_scan(void *ctx, scan_buffer_t *buf) {
    mux_adc_scan_ctx_t *c = (mux_adc_scan_ctx_t *)ctx;
    buf->count = c->total_channels;

    for (uint8_t ch = 0; ch < c->total_channels; ch++) {
        // 设置 MUX 地址选择引脚
        for (uint8_t s = 0; s < c->sel_pin_count; s++) {
            const gpio_level_t level = (ch >> s) & 1u ? GPIO_HIGH : GPIO_LOW;
            c->gpio->write(c->sel_pins[s], level);
        }
        // 等待 MUX 输出建立
        drv_delay_us(c->settle_us);
        // 读取 ADC
        uint16_t raw = 0;
        c->adc->read(c->adc_channel, &raw);
        buf->values[ch] = raw;
    }
}
```

### 4.6 buf_btn_source — 缓冲区按键输入源

```c
// input/buf_btn_source.h
#pragma once
#include "drv_common.h"
#include "scan_buffer.h"

// 按键输入源统一接口
typedef struct {
    drv_status_t (*read_buttons)(void *ctx, uint16_t *button_mask);
    void *ctx;
    uint8_t button_count;
} button_source_t;

// 基于 scan_buffer 的按键输入源上下文
// 每个按键占独立 MUX 通道，5V→2.5V 分压，按下时读数 ≥ threshold
typedef struct {
    const scan_buffer_t *buf;
    uint8_t    channel_indices[12];  // 各按键对应的 scan_buffer 通道索引
    uint16_t   threshold;            // 判定为按下的 ADC 阈值（对应 ~2.5V）
    uint8_t    count;                // 按键数量
} buf_btn_ctx_t;

static inline drv_status_t buf_btn_read(void *ctx, uint16_t *mask) {
    const buf_btn_ctx_t *c = (const buf_btn_ctx_t *)ctx;
    *mask = 0;
    for (uint8_t i = 0; i < c->count; i++) {
        if (c->buf->values[c->channel_indices[i]] >= c->threshold) {
            *mask |= (1u << i);
        }
    }
    return DRV_OK;
}
```

阈值检测方式天然支持**多键同按**：每个按键在独立 MUX 通道上有独立的分压电路，互不干扰。

## 五、信号处理管线架构（v4 改进：统一为显式 pipeline，删除泛型 chain）

### 5.1 内部信号表示：无符号原始范围，不归一化

信号处理管线**不做归一化**，全程保持传感器原始范围。值域转换由独立的 `output_adapter` 层在协议打包前一次性完成（见第六章）。

| 传感器 | 原始范围 | 中心值 | 数据类型 |
|--------|---------|--------|---------|
| 12-bit ADC | 0–4095 | ~2048（校准值） | uint16_t |
| 14-bit Hall | 0–16383 | ~8192（校准值） | uint16_t |

**为什么不在管线内归一化：**

- **精度损失是真实的** — 14-bit 传感器抖动 ±2~3 LSB，一次整数除法映射可能把信号淹没在量化噪声中
- **校准不对称时引入非线性** — 正/负向量程不同，归一化会在零点附近产生增益跳变
- **管线各阶段不需要统一量纲** — 故障检测在 `uint16_t` 域，滤波/死区在 `int32_t` 域围绕零点操作，天然不关心绝对范围

**v4 类型域分界：**

```
              uint16_t 域                    int32_t 域
         ┌──────────────────┐    ┌──────────────────────────┐
  raw ──→│ fault_detect     │───→│ calibrate (分界点)        │
         │ 阈值比较          │    │ linear: raw - center     │
         └──────────────────┘    │ circular: wrap_subtract  │
                                 ├──────────────────────────┤
                                 │ lowpass_filter           │
                                 ├──────────────────────────┤
                                 │ deadzone                 │
                                 ├──────────────────────────┤
                                 │ output (有符号偏移量)     │
                                 └──────────────────────────┘
                                          │
                                 output_adapter 按需缩放（唯一映射点）
```

### 5.2 设计动机

原设计中 `joystick_core.c` 的 `process_axis()` 函数同时负责 ADC 采集、故障检测、校准计算、死区处理、按键消抖五个职责。任何一个需求变化（换消抖算法、换死区曲线、加数字滤波）都要修改这个文件。

改进：拆分为信号处理管线（pipeline），每个阶段是独立模块，各自只有一个变更理由。

> **v4 变更：** v3 同时存在泛型 `signal_chain_t`（`int16_t` 统一签名）和显式 `axis_pipeline_t`（`uint16_t→int32_t` 分界）两套接口。v4 删除泛型 chain，只保留显式 pipeline。理由：管线阶段顺序固定，真正变化的是参数和策略，泛型 chain 的灵活性不被需要，反而引入类型不匹配和概念混淆。

### 5.3 各阶段独立实现（v4 统一签名）

```c
// signal/calibrator.c
int32_t calibrate_linear(uint16_t raw, const calibrator_ctx_t *ctx) {
    // 不归一化，直接返回有符号偏移量
    return (int32_t)raw - (int32_t)ctx->center;
}

int32_t calibrate_circular(uint16_t raw, const calibrator_ctx_t *ctx) {
    // 环形传感器（霍尔角度）：处理回绕
    int32_t diff = (int32_t)raw - (int32_t)ctx->center;
    int32_t half_range = (int32_t)(ctx->max - ctx->min) / 2;
    if (diff > half_range)  diff -= (int32_t)(ctx->max - ctx->min);
    if (diff < -half_range) diff += (int32_t)(ctx->max - ctx->min);
    return diff;
}
```

### 5.4 显式信号管线（v4 唯一的信号链接口）

管线阶段顺序在产品中基本固定，真正变化的是每个阶段的**参数和策略**。用显式函数实现，类型分界清晰：

```c
// signal/axis_pipeline.h
#pragma once
#include <stdint.h>
#include "fault_detector.h"
#include "calibrator.h"
#include "filter.h"
#include "deadzone.h"

typedef struct {
    uint16_t       fault_threshold_low;
    uint16_t       fault_threshold_high;
    uint8_t        filter_alpha;       // 0 = 不启用滤波
    int32_t        deadzone_threshold;

    // 校准上下文

    calibrate_fn_t     calibrate;     // 线性 / 环形（产品配置注入）
    // 滤波上下文

    // 死区

    deadzone_fn_t      dz_fn;
    // 输出
    int32_t            output;        // 有符号偏移量（不归一化）

} axis_pipeline_t;
```

```c
// signal/axis_pipeline.c
void axis_pipeline_init(axis_pipeline_t *p,
                        const axis_pipeline_config_t *cfg,
                        calibrate_fn_t calibrate,
                        uint8_t resolution_bits) {
    p->fault_ctx.threshold_low  = cfg->fault_threshold_low;
    p->fault_ctx.threshold_high = cfg->fault_threshold_high;
    p->calibrate    = calibrate;
    p->filter_ctx.alpha = cfg->filter_alpha;
    p->filter_ctx.prev_output = 0;
    p->dz_threshold = cfg->deadzone_threshold;
    p->dz_fn        = cfg->deadzone_fn;
    p->output       = 0;
    p->max_deflection = (uint16_t)(1u << (resolution_bits - 1));  // range/2
}

void axis_pipeline_process(axis_pipeline_t *p, uint16_t raw) {
    // 1. 故障检测（uint16_t 域）
    if (fault_detect(&p->fault_ctx, raw)) {
        p->output = 0;
        return;
    }

    // 2. 校准（uint16_t → int32_t，类型分界点）
    //    线性传感器: (int32_t)raw - center
    //    数字角度:   circular_subtract(raw, center)
    int32_t val = p->calibrate(raw, &p->cal_ctx);

    // 3. 低通滤波（int32_t 域）
    if (p->filter_ctx.alpha > 0) {
        val = lowpass_filter(&p->filter_ctx, val);
    }

    // 4. 死区（int32_t 域，围绕零点）
    val = p->dz_fn(val, p->dz_threshold);

    p->output = val;
}
```

## 六、协议层架构（v4 改进：Output Adapter + Profile 驱动 + Protocol Bundle）

### 6.1 设计动机

v3 协议层存在的问题：

1. **`void *config`** — `init(const void *config)` 无类型检查，传错 config 编译通过、运行崩溃 → v3 已用 Protocol Bundle 解决
2. **协议行为 hardcode** — OD 索引、PGN、字节打包、值范围全写死在代码里 → v3 已用 Profile 驱动解决
3. **值域转换嵌入协议引擎** — `canopen_engine_send()` 同时做值缩放和协议打包，违反 SRP → **v4 新增 Output Adapter 层解决**
4. **`joystick_data_t` 非自描述** — `max_deflection` 为单一值，假设所有轴分辨率相同，但 JS-300 混合 12-bit/14-bit → **v4 改为 per-axis**

### 6.3 轴输出映射描述（通用）

所有协议共享的轴值转换描述，定义目标范围和方向：

```c
// output/axis_output_map.h — v4: 从 protocols/ 移至 output/，供 output_adapter 使用
#pragma once
#include <stdint.h>

typedef struct {
    int32_t output_min;     // 目标最小值（如 -10000 / 0）
    int32_t output_max;     // 目标最大值（如 +10000 / 65535）
    int8_t  direction;      // +1 正常, -1 反向
} axis_output_map_t;

// 通用转换：管线输出 → 协议目标范围（纯函数）
static inline int32_t axis_map_output(int32_t val, int32_t max_deflection,
                                       const axis_output_map_t *map) {
    int32_t center = (map->output_min + map->output_max) / 2;
    int32_t half   = (map->output_max - map->output_min) / 2;
    return center + val * map->direction * half / max_deflection;
}
```

### 6.4 Output Adapter 层（v4 新增）

**设计动机：** v3 中值域转换嵌入协议引擎内部，导致：
- 协议引擎违反 SRP（转换 + 打包两个职责）
- HMI 显示/数据记录/远程调试想用缩放后的值无法复用
- 测试值转换需要 mock CAN 驱动
- `joystick_data_t` 非自描述（`axes[]` 的含义依赖外部上下文）

v4 引入 `output_adapter` 层，职责单一：将管线内部偏移量转换为自描述的输出值。

```c
// output/output_adapter.h — v4 新增：值域转换层
#pragma once
#include "joystick_data.h"
#include "axis_output_map.h"

// 每轴输出描述：自描述，不依赖外部上下文
typedef struct {
    int32_t  value;          // 转换后的值（目标范围内）
    int32_t  output_min;     // 该值的有效下界
    int32_t  output_max;     // 该值的有效上界
    uint8_t  valid;          // 数据是否有效
} axis_output_t;

typedef struct {
    axis_output_t axes[MAX_AXES];
    uint8_t       axes_count;
    uint16_t      buttons;
    uint8_t       buttons_count;
    uint32_t      timestamp_ms;
    uint8_t       safety_state;
    uint8_t       error_flags;
} output_data_t;

// 纯函数：joystick_data_t → output_data_t
void output_adapt(const joystick_data_t *raw,
                  const axis_output_map_t *maps,
                  uint8_t map_count,
                  output_data_t *out);
```

```c
// output/output_adapter.c
#include "output_adapter.h"

void output_adapt(const joystick_data_t *raw,
                  const axis_output_map_t *maps,
                  uint8_t map_count,
                  output_data_t *out) {
    out->axes_count    = raw->axes_count;
    out->buttons       = raw->buttons;
    out->buttons_count = raw->buttons_count;
    out->timestamp_ms  = raw->timestamp_ms;
    out->safety_state  = raw->safety_state;
    out->error_flags   = raw->error_flags;

    for (uint8_t i = 0; i < map_count && i < raw->axes_count; i++) {
        const axis_output_map_t *m = &maps[i];
        out->axes[i].value = axis_map_output(
            raw->axes[i], (int32_t)raw->max_deflection[i], m);
        out->axes[i].output_min = m->output_min;
        out->axes[i].output_max = m->output_max;
        out->axes[i].valid      = (raw->axes_valid >> i) & 1u;
    }
}
```

**层级职责对照（v4）：**

```
层                    输入类型         输出类型         职责
─────────────────────────────────────────────────────────────
axis_pipeline         uint16_t raw     int32_t offset   信号处理（故障/校准/滤波/死区）
joystick_data_t       —                —                管线内部数据合同（原始偏移量）
output_adapter        joystick_data_t  output_data_t    值域转换（纯函数，可复用）
protocol_engine       output_data_t    CAN frame        协议打包（OD/PGN/raw frame）
```

**设计模式对照：**

| 模式 | v3 | v4 |
|------|------|--------|
| **SRP** | 协议层兼做转换+打包 | 转换和打包分离 |
| **Adapter** | 转换逻辑嵌入协议引擎内部 | `output_adapter` 是独立 Adapter |
| **Anti-Corruption Layer** | 协议层直接解释管线内部表示 | `output_data_t` 是自描述的边界对象 |
| **可复用性** | HMI/日志/调试想用缩放后的值需重复代码 | `output_adapt()` 是纯函数，任何消费者可调用 |
| **可测试性** | 测试值转换需要 mock CAN 驱动 | `output_adapt()` 纯函数，直接单元测试 |

### 6.6 CANopen 通用引擎（v4 改进：只管打包，不做值转换）

```c
// protocols/canopen/canopen_engine.c
#include "canopen_profile.h"
#include "canopen_stack.h"
#include "output_adapter.h"     // v4: 接收 output_data_t

static const canopen_profile_t *profile;
static const can_driver_t *can;
static uint8_t node_id;

drv_status_t canopen_engine_init(const canopen_config_t *cfg) {
    can = cfg->can;
    node_id = cfg->node_id;
    profile = cfg->profile;

    co_node_init(node_id);
    co_nmt_set_heartbeat(profile->heartbeat_ms);

    // 根据 profile 初始化 OD 默认值
    od_load_defaults(profile->od);

    // 根据 profile 配置 TPDO 映射
    for (uint8_t t = 0; t < profile->tpdo_count; t++) {
        const tpdo_config_t *tp = &profile->tpdos[t];
        for (uint8_t e = 0; e < tp->count; e++) {
            co_tpdo_map(t + 1, tp->entries[e].od_index,
                        tp->entries[e].od_subindex,
                        tp->entries[e].bit_length);
        }
        co_tpdo_set_event_timer(t + 1, tp->event_timer_ms);
    }

    event_bus_subscribe(EVT_SAFETY_STATE_CHANGED, on_safety_changed);
    return DRV_OK;
}

// v4: 接收 output_data_t（已经过 output_adapter 转换，值已在目标范围内）
drv_status_t canopen_engine_send(const output_data_t *data) {
    // 直接将已转换的值写入 OD，不再调用 axis_map_output()
    for (uint8_t i = 0; i < profile->tpdo_axis_count && i < data->axes_count; i++) {
        od_update_value(profile->od,
                        profile->tpdos[0].entries[i].od_index,
                        profile->tpdos[0].entries[i].od_subindex,
                        data->axes[i].value);
    }
    // 按键写入 OD
    od_update_value(profile->od, profile->digital_input_index,
                    profile->digital_input_subindex, data->buttons);
    return DRV_OK;
}

drv_status_t canopen_engine_process(void) {
    co_node_process();
    return DRV_OK;
}

static void on_safety_changed(const event_t *evt) {
    if (evt->data.safety_state == SAFETY_STOP) {
        co_emcy_send(profile->emcy_code_sensor_fault);
    }
}
```

### 6.7 J1939 Profile — PGN + SPN 描述符

J1939 没有 OD，变化点是 PGN 选择和帧内字节打包方式。用 SPN 描述符定义：

```c
// protocols/j1939/j1939_profile.h
#pragma once
#include <stdint.h>

typedef enum {
    SPN_SOURCE_AXIS,       // 来源：轴数据
    SPN_SOURCE_BUTTON,     // 来源：按键位掩码
    SPN_SOURCE_SAFETY,     // 来源：安全状态
    SPN_SOURCE_CONST,      // 来源：固定值（如 0xFF 填充）
} spn_source_t;

typedef struct {
    spn_source_t source;
    uint8_t  source_index;    // 轴/按键编号（CONST 时为固定值）
    uint8_t  byte_offset;     // 帧中起始字节（0-7）
    uint8_t  bit_offset;      // 字节内起始位（0-7）
    uint8_t  bit_length;      // 位宽
    int32_t  scale_num;       // 缩放分子
    int32_t  scale_den;       // 缩放分母（physical = raw * num / den + offset）
    int32_t  add_offset;      // 偏移量
    int8_t   direction;       // +1 正常, -1 反向
} spn_descriptor_t;

typedef struct {
    uint32_t pgn;
    uint8_t  priority;
    uint16_t tx_interval_ms;
    const spn_descriptor_t *spns;
    uint8_t  spn_count;
} pgn_descriptor_t;

typedef struct {
    const char             *profile_name;
    uint8_t                 source_address;
    const pgn_descriptor_t *pgns;
    uint8_t                 pgn_count;
} j1939_profile_t;
```

### 6.8 J1939 通用引擎（v4 改进：接收 output_data_t）

```c
// protocols/j1939/j1939_engine.c
#include "j1939_profile.h"
#include "output_adapter.h"     // v4: 接收 output_data_t

static const j1939_profile_t *profile;
static const can_driver_t *can;

drv_status_t j1939_engine_init(const j1939_config_t *cfg) {
    can = cfg->can;
    profile = cfg->profile;
    return DRV_OK;
}

// v4: 接收 output_data_t，轴值已经过 output_adapter 转换
drv_status_t j1939_engine_send(const output_data_t *data) {
    for (uint8_t p = 0; p < profile->pgn_count; p++) {
        const pgn_descriptor_t *pgn = &profile->pgns[p];
        can_frame_t frame = {0};
        frame.id = ((uint32_t)pgn->priority << 26)
                 | (pgn->pgn << 8)
                 | profile->source_address;
        frame.is_extended = 1;
        frame.dlc = 8;
        memset(frame.data, 0xFF, 8);

        for (uint8_t s = 0; s < pgn->spn_count; s++) {
            const spn_descriptor_t *spn = &pgn->spns[s];
            int32_t val = spn_get_source(data, spn);
            // J1939 SPN 仍需自己的 scale/offset 打包（协议特有的位级编码）
            uint32_t packed = (uint32_t)(val * spn->direction
                              * spn->scale_num / spn->scale_den
                              + spn->add_offset);
            spn_write_bits(frame.data, packed,
                           spn->byte_offset, spn->bit_offset, spn->bit_length);
        }
        can->send(&frame);
    }
    return DRV_OK;
}

static int32_t spn_get_source(const output_data_t *data,
                              const spn_descriptor_t *spn) {
    switch (spn->source) {
        case SPN_SOURCE_AXIS:    return data->axes[spn->source_index].value;
        case SPN_SOURCE_BUTTON:  return data->buttons;
        case SPN_SOURCE_SAFETY:  return data->safety_state;
        case SPN_SOURCE_CONST:   return spn->source_index;
        default:                 return 0;
    }
}
```

### 6.10 协议能力与运行时接口（v4 改进：send 接收 output_data_t）

能力位掩码保留——运行时查询"支不支持诊断"仍然有用：

```c
// protocols/protocol_runtime.h
#include "output_adapter.h"     // v4: send 接收 output_data_t

typedef enum {
    PROTO_CAP_RECEIVABLE    = (1 << 0),
    PROTO_CAP_CONFIGURABLE  = (1 << 1),
    PROTO_CAP_DIAGNOSABLE   = (1 << 2),
    PROTO_CAP_SAFETY        = (1 << 3),
} protocol_capability_t;

typedef struct {
    protocol_type_t type;
    uint32_t capabilities;

    drv_status_t (*send)(const output_data_t *data);   // v4: 接收自描述数据
    drv_status_t (*process)(void);
    drv_status_t (*get_bus_diag)(protocol_status_t *status);
} protocol_runtime_t;
```

初始化时根据 bundle type 构建运行时接口（唯一的 switch，启动时执行一次）：

```c
// protocols/protocol_init.c
static protocol_runtime_t runtime;

drv_status_t protocol_init(const protocol_bundle_t *bundle) {
    runtime.type = bundle->type;

    switch (bundle->type) {
        case PROTOCOL_TYPE_CANOPEN:
            canopen_engine_init(&bundle->canopen);
            runtime.send         = canopen_engine_send;
            runtime.process      = canopen_engine_process;
            runtime.get_bus_diag = canopen_engine_get_diag;
            runtime.capabilities = PROTO_CAP_RECEIVABLE
                                 | PROTO_CAP_CONFIGURABLE
                                 | PROTO_CAP_DIAGNOSABLE;
            break;

        case PROTOCOL_TYPE_J1939:
            j1939_engine_init(&bundle->j1939);
            runtime.send         = j1939_engine_send;
            runtime.process      = j1939_engine_process;
            runtime.get_bus_diag = NULL;
            runtime.capabilities = PROTO_CAP_RECEIVABLE;
            break;

        case PROTOCOL_TYPE_CAN_RAW:
            can_raw_engine_init(&bundle->can_raw);
            runtime.send         = can_raw_engine_send;
            runtime.process      = can_raw_engine_process;
            runtime.get_bus_diag = NULL;
            runtime.capabilities = PROTO_CAP_RECEIVABLE;
            break;

        default:
            return DRV_ERR_INVALID;
    }
    return DRV_OK;
}

const protocol_runtime_t* protocol_get_runtime(void) {
    return &runtime;
}
```

运行时使用（v4：main_task 中先经 output_adapter 转换，再传给协议引擎）：

```c
void app_run_cycle(const joystick_data_t *data,
                   const axis_output_map_t *maps, uint8_t map_count) {
    const protocol_runtime_t *proto = protocol_get_runtime();

    // v4: output_adapter 转换（纯函数，可被 HMI/日志等复用）
    output_data_t out;
    output_adapt(data, maps, map_count, &out);

    proto->send(&out);

    if (proto->capabilities & PROTO_CAP_RECEIVABLE) {
        proto->process();
    }
    if (proto->capabilities & PROTO_CAP_DIAGNOSABLE) {
        protocol_status_t status;
        proto->get_bus_diag(&status);
    }
}
```

## 八、事件总线（观察者模式）

### 8.1 设计动机

原设计中协议插件和安全监控之间没有通信机制。实际场景需求：
- CAN Bus-Off 发生时 → 安全监控需要被通知
- 安全状态变化时 → 协议层需要发送紧急消息
- 传感器故障时 → 诊断模块需要记录日志

事件总线提供了松耦合的发布/订阅机制。

### 8.2 事件总线接口

```c
// event/event_bus.h
#pragma once
#include <stdint.h>

typedef enum {
    EVT_SENSOR_FAULT,
    EVT_CAN_BUS_OFF,
    EVT_CAN_RECOVERED,
    EVT_SAFETY_STATE_CHANGED,
    EVT_CALIBRATION_COMPLETE,
    EVT_WATCHDOG_WARNING,
    EVT_DEVICE_STATE_CHANGED,
    EVT_MAX
} event_type_t;

typedef struct {
    event_type_t type;
    uint32_t     timestamp;
    union {
        uint8_t  sensor_id;
        uint8_t  safety_state;  // 0=normal, 1=degraded, 2=safe-stop
        uint32_t error_code;
        uint8_t  new_device_state;
    } data;
} event_t;

typedef void (*event_handler_t)(const event_t *evt);

#define EVENT_BUS_MAX_SUBSCRIBERS 4  // 每个事件类型最多 4 个订阅者

void event_bus_init(void);
void event_bus_subscribe(event_type_t type, event_handler_t handler);
void event_bus_publish(const event_t *evt);
```

```c
// event/event_bus.c
#include "event_bus.h"

static struct {
    event_handler_t handlers[EVENT_BUS_MAX_SUBSCRIBERS];
    uint8_t count;
} subscribers[EVT_MAX];

void event_bus_init(void) {
    for (uint8_t i = 0; i < EVT_MAX; i++) {
        subscribers[i].count = 0;
    }
}

void event_bus_subscribe(event_type_t type, event_handler_t handler) {
    if (type < EVT_MAX && subscribers[type].count < EVENT_BUS_MAX_SUBSCRIBERS) {
        subscribers[type].handlers[subscribers[type].count++] = handler;
    }
}

void event_bus_publish(const event_t *evt) {
    if (evt->type < EVT_MAX) {
        for (uint8_t i = 0; i < subscribers[evt->type].count; i++) {
            subscribers[evt->type].handlers[i](evt);
        }
    }
}
```

## 九、设备状态机

### 9.1 设计动机

原设计缺少明确的设备生命周期状态管理。实际嵌入式产品需要管理启动自检、预操作、正常运行、安全停止、校准、固件更新等状态。

### 9.2 状态机定义

```c
// app/device_state.h
#pragma once
#include "event_bus.h"

typedef enum {
    STATE_BOOT,            // 启动自检
    STATE_PRE_OPERATIONAL, // 初始化完成，等待使能
    STATE_OPERATIONAL,     // 正常工作
    STATE_SAFE_STOP,       // 安全停止（故障触发）
    STATE_CALIBRATION,     // 校准模式
    STATE_FIRMWARE_UPDATE, // 固件更新
    STATE_MAX
} device_state_t;

typedef struct {
    device_state_t current;
    drv_status_t (*on_enter)(void);
    drv_status_t (*on_execute)(void);
    drv_status_t (*on_exit)(void);

    // 状态转移表
    struct {
        event_type_t   trigger;
        device_state_t next;
        bool (*guard)(void);  // 转移守卫条件
    } transitions[8];
    uint8_t transition_count;
} state_handler_t;

void device_state_init(void);
void device_state_process(void);
device_state_t device_state_get_current(void);
```

```c
// app/device_state.c

static state_handler_t state_table[STATE_MAX];
static device_state_t  current_state = STATE_BOOT;

void device_state_init(void) {
    // 注册状态处理器
    state_table[STATE_BOOT] = (state_handler_t){
        .on_enter   = boot_self_test,
        .on_execute = boot_check_complete,
        .transitions = {
            { .trigger = EVT_DEVICE_STATE_CHANGED, .next = STATE_PRE_OPERATIONAL,
              .guard = boot_test_passed },
            { .trigger = EVT_SENSOR_FAULT, .next = STATE_SAFE_STOP },
        },
        .transition_count = 2,
    };

    state_table[STATE_OPERATIONAL] = (state_handler_t){
        .on_enter   = operational_enter,
        .on_execute = operational_run,
        .on_exit    = operational_exit,
        .transitions = {
            { .trigger = EVT_SAFETY_STATE_CHANGED, .next = STATE_SAFE_STOP },
            { .trigger = EVT_CALIBRATION_COMPLETE, .next = STATE_CALIBRATION,
              .guard = calibration_requested },
        },
        .transition_count = 2,
    };

    // ... 其他状态配置

    // 订阅所有可能触发转移的事件
    event_bus_subscribe(EVT_SAFETY_STATE_CHANGED, on_state_event);
    event_bus_subscribe(EVT_SENSOR_FAULT, on_state_event);
    event_bus_subscribe(EVT_DEVICE_STATE_CHANGED, on_state_event);
}

static void on_state_event(const event_t *evt) {
    state_handler_t *handler = &state_table[current_state];
    for (uint8_t i = 0; i < handler->transition_count; i++) {
        if (handler->transitions[i].trigger == evt->type) {
            if (!handler->transitions[i].guard
                || handler->transitions[i].guard()) {
                // 执行状态转移
                if (handler->on_exit) handler->on_exit();
                current_state = handler->transitions[i].next;
                state_handler_t *next = &state_table[current_state];
                if (next->on_enter) next->on_enter();
                break;
            }
        }
    }
}

void device_state_process(void) {
    if (state_table[current_state].on_execute) {
        state_table[current_state].on_execute();
    }
}
```

## 十一、RTOS 任务架构

### 11.1 设计动机

工业摇杆固件必须使用 RTOS，原因：

| 需求 | 裸循环问题 | RTOS 解决方案 |
|------|-----------|--------------|
| 安全监控 2ms 响应 | 被协议处理 or 信号链阻塞 | 高优先级任务抢占 |
| CANopen 心跳 1ms 精度 | 主循环抖动大 | 独立任务 + 硬件定时器 |
| 看门狗喂狗时序 | 长处理链导致超时 | 独立高优先级任务 |
| CAN 接收实时性 | 轮询方式丢帧 | ISR + 队列 + 任务 |

### 11.2 RTOS 抽象层 — 采用 CMSIS-RTOS v2

不自研抽象层，直接使用 ARM 官方 **CMSIS-RTOS v2** 标准 API。原因：

| 对比项 | 自研 os_abstraction.h | CMSIS-RTOS v2 |
|--------|----------------------|---------------|
| API 覆盖 | 需手动实现每个 RTOS 适配 | FreeRTOS / ThreadX / RTX5 均有官方适配 |
| ISR 安全 | 需手动区分 `_from_isr` 版本 | 自动检测 IPSR 寄存器，同一 API 通用 |
| 绝对延时 | 未定义（导致周期漂移） | `osDelayUntil()` 已内置 |
| 优先级继承 | 需额外约定 | `osMutexNew()` 直接传 `osMutexPrioInherit` |
| 平台兼容 | STM32 / S32K 需各写一套 | STM32 CubeMX 默认生成；S32K FreeRTOS 直接支持 |
| 维护成本 | 自维护 | ARM 官方维护，零成本 |
| MISRA 合规 | 需自行保证 | CMSIS 头文件已通过 MISRA 扫描 |

项目中使用的核心 API：

```c
#include "cmsis_os2.h"

// ---- 任务管理 ----
osThreadId_t osThreadNew(osThreadFunc_t func, void *argument,
                          const osThreadAttr_t *attr);

// ---- 互斥量（支持优先级继承）----
osMutexId_t osMutexNew(const osMutexAttr_t *attr);  // attr 含 osMutexPrioInherit

// ---- 消息队列（ISR/任务 统一 API）----
osMessageQueueId_t osMessageQueueNew(uint32_t msg_count, uint32_t msg_size,
                                      const osMessageQueueAttr_t *attr);
osStatus_t osMessageQueuePut(osMessageQueueId_t mq_id, const void *msg_ptr,
                              uint8_t msg_prio, uint32_t timeout);
osStatus_t osMessageQueueGet(osMessageQueueId_t mq_id, void *msg_ptr,
                              uint8_t *msg_prio, uint32_t timeout);

// ---- 定时与延时 ----
osStatus_t osDelay(uint32_t ticks);
```

FreeRTOS 适配层由 `CMSIS/RTOS2/FreeRTOS/` 目录提供（CubeMX 自动包含），无需自行编写。

### 11.3 任务分解（v4 改进：safety_task 升级为 L2 独立通道）

```

│                     RTOS 任务架构 (v4)                                │
├────────────┬──────────┬────────┬─────────────────────────────────────┤
```

### 11.5 各任务职责（v4 重构）

```c
// safety_task — v4: 升级为 E-Gas Level 2 独立计算通道
// 规则：禁止使用 mutex，只通过 atomic/volatile 读取共享数据
static void safety_task_entry(void *param) {
    safety_channel_ctx_t *l2_ch  = app_get_l2_channel();
    safety_comparator_t  *l2_cmp = app_get_l2_comparator();
    safety_wdg_ctx_t     *l2_wdg = app_get_l2_wdg();
    uint32_t prev_tick = osKernelGetTickCount();

    for (;;) {
        // ── Level 2: 独立传感器采集 + 简化计算 ──
        safety_channel_output_t l2_out;
        safety_channel_compute(l2_ch, &l2_out);

        // ── Level 2: 读取 Level 1 输出并比对 ──
        int32_t l1_axes[MAX_AXES];
        safety_shared_read_l1_output(l1_axes);  // atomic 读取

        compare_result_t cmp = safety_compare(
            l2_cmp, l1_axes, &l2_out, l2_ch->axes_count);

        switch (cmp) {
            case COMPARE_OK:
                safety_wdg_update_token(l2_wdg);   // 允许喂狗
                break;
            case COMPARE_DEGRADED:
                event_bus_publish(&(event_t){
                    .type = EVT_SAFETY_STATE_CHANGED,
                    .data.safety_state = 1 });       // 降级
                break;
            case COMPARE_FAIL:
                safe_shutdown_activate();             // L3: 直接硬件关断
                break;
        }

        // ── Level 3: 条件喂狗 ──
        safety_wdg_service(l2_wdg);

        // ── 附加诊断（调用平台认证安全库）──
        safety_diagnostics();          // CPU/RAM/Flash/Clock 自检（X-CUBE-STL / NXP Safety SW）

        // ── 设备状态机 + ISR 事件分发 ──
        device_state_process();
        event_bus_dispatch_deferred();

        osDelayUntil(prev_tick += 2);  // 绝对延时，保证恒定 2ms 周期
    }
}

// main_task — Level 1 功能通道（v4: 增加 output_adapter + 程序流标记）
static void main_task_entry(void *param) {
    joystick_t *js           = app_get_joystick();
    board_scanner_t *scanner = app_get_scanner();
    scan_buffer_t   *buf     = app_get_scan_buf();
    const protocol_runtime_t *proto = protocol_get_runtime();
    joystick_data_t data;
    output_data_t   out;
    uint32_t prev_tick = osKernelGetTickCount();

    for (;;) {
        cf_marker_checkpoint(CF_MAIN_START);     // v4: 程序流标记

        if (device_state_get_current() == STATE_OPERATIONAL) {
            // 1. 板级扫描器一次性采集所有通道 → scan_buffer
            scanner->scan(scanner->ctx, buf);
            cf_marker_checkpoint(CF_MAIN_SCAN_DONE);

            // 2. 摇杆更新：buf_axis_source / buf_btn_source 从 buffer 读取
            joystick_update(js);
            joystick_get_data(js, &data);
            cf_marker_checkpoint(CF_MAIN_SIGNAL_DONE);

            // 3. v4: 写入共享比较区（供 Level 2 比对）
            safety_shared_write_l1_output(data.axes, data.axes_count);

            // 4. v4: output_adapter 转换（纯函数）
            output_adapt(&data, product_config.output_maps,
                         product_config.output_map_count, &out);

            // 5. mutex 保护发送（设超时，避免死锁）
            if (osMutexAcquire(g_data_mutex, 5) == osOK) {
                proto->send(&out);
                osMutexRelease(g_data_mutex);
            } else {
                event_bus_publish(&(event_t){ .type = EVT_MUTEX_TIMEOUT });
            }
        }

        cf_marker_checkpoint(CF_MAIN_END);
        osDelayUntil(prev_tick += 10);  // 绝对延时，保证恒定 10ms 周期
    }
}

// proto_task — 协议栈实时处理（v4 无变化）
static void proto_task_entry(void *param) {
    const protocol_runtime_t *proto = protocol_get_runtime();
    uint32_t prev_tick = osKernelGetTickCount();

    for (;;) {
        proto->process();  // CANopen: NMT、SDO、心跳、RPDO
        osDelayUntil(prev_tick += 1);  // 绝对延时，保证恒定 1ms 周期
    }
}

// diag_task — 低优先级后台服务（v4 无变化）
static void diag_task_entry(void *param) {
    uint32_t prev_tick = osKernelGetTickCount();

    for (;;) {
        diagnostics_update();
        nv_storage_flush();   // 将缓存写入 Flash/EEPROM
        logging_flush();
        task_health_check();  // 检查各任务 alive 标记
        osDelayUntil(prev_tick += 100);  // 绝对延时，保证恒定 100ms 周期
    }
}
```

### 11.7 事件总线的 ISR 安全

原有 `event_bus_publish()` 在回调中直接调用 handler，不适合从 ISR 发布。改进为**延迟分发**模式：

```c
// event/event_bus.h — ISR 安全版本

// 任务上下文：立即同步分发（原有行为）
void event_bus_publish(const event_t *evt);

// ISR 上下文：入队列，由 safety_task 延迟分发
// 内部使用 osMessageQueuePut()，CMSIS-RTOS v2 自动检测 ISR 上下文
void event_bus_publish_from_isr(const event_t *evt);

// 在 safety_task 中周期调用，处理 ISR 投递的事件
void event_bus_dispatch_deferred(void);
```

> **注意**：CMSIS-RTOS v2 的 `osMessageQueuePut()` 会通过 ARM Cortex-M 的 IPSR 寄存器自动判断当前是否在 ISR 上下文，内部自动选择 `xQueueSendFromISR()` 或 `xQueueSend()`。因此 `event_bus_publish_from_isr()` 的实现无需手动区分上下文。

### 11.8 时序要求与优先级反转防范

```
时间线（最坏情况）：

0ms ─── safety_task 开始执行 (2ms 周期)
         安全监控 + 看门狗喂狗
0.3ms ── safety_task 完成，让出 CPU
         proto_task 获得 CPU (1ms 周期)
         CANopen 协议栈处理
0.8ms ── proto_task 完成
         main_task 获得 CPU (10ms 周期)
         摇杆采集 + 信号链 + 协议发送
2ms ──── safety_task 再次抢占 main_task
         确保安全监控不被阻塞
```

关键设计规则：
- `safety_task` 中**禁止使用 mutex**（避免优先级反转导致安全超时）
- 安全任务只读取数据，使用 atomic 或 volatile 访问
- `protocol_status` 改为 atomic 单写者（proto_task 写，safety_task 无锁读）
- CAN RX 中断处理时间 < 5μs（只做 `osMessageQueuePut` 入队列）
- 所有 mutex 创建时指定 `osMutexPrioInherit`（优先级继承）
- `main_task` 的 mutex 获取设置 5ms 超时（避免死锁）
- 所有周期任务使用 `osDelayUntil()` 绝对延时（避免周期漂移）

## 十二、功能安全架构（v4 新增：E-Gas 三层模型，SIL 2 / PL d）

### 12.1 安全需求与标准对照

工业摇杆用于工程机械、农业设备等安全关键场景，需满足：

| 标准 | 等级 | 核心要求 |
|------|------|---------|
| IEC 61508 | SIL 2 | DC ≥ 90%，安全功能与控制功能独立，程序流监控 |
| ISO 13849 | PL d, Cat.3 | 双通道结构，单一故障不丧失安全功能 |

### 12.2 为什么 v3 不满足

v3 的 `safety_task` 是"旁观者"——高优先级执行，但与 `main_task` 共享 RTOS、内存空间、ADC 驱动、事件总线。**优先级隔离 ≠ 独立性**。RTOS 内核 bug、堆栈溢出、系统节拍丢失都是共因失效点。

### 12.4 Level 2 — 独立安全计算通道

**核心原则：代码独立、数据独立、执行独立。**

Level 2 的信号计算**必须独立编写**，不能复用 Level 1 的 `axis_pipeline` 代码。IEC 61508-3 7.4.2.2 要求软件多样性——即使算法相同，也应独立实现以避免系统性故障。

```c
// safety/safety_channel.h — Level 2 独立计算通道
#pragma once
#include <stdint.h>
#include "drv_common.h"
#include "adc_voltage_reader.h"

#define SAFETY_MV_LOW   100    // 开路检测下限 (mV)
#define SAFETY_MV_HIGH  3200   // 短路检测上限 (mV)

typedef struct {
    const adc_voltage_reader_t *adc;
    uint8_t   channel_indices[MAX_AXES];
    uint8_t   axes_count;
    uint16_t  center[MAX_AXES];    // L2 独立的校准参数（独立存储区）
} safety_channel_ctx_t;

typedef struct {
    int32_t axes[MAX_AXES];
    uint8_t axes_count;
    uint8_t sensor_ok;             // 独立的传感器诊断
} safety_channel_output_t;

void safety_channel_init(safety_channel_ctx_t *ctx,
                          const safety_channel_init_cfg_t *cfg);
void safety_channel_compute(safety_channel_ctx_t *ctx,
                             safety_channel_output_t *out);
```

```c
// safety/safety_channel.c — 独立实现，不调用 signal chain 的任何函数
#include "safety_channel.h"

void safety_channel_compute(safety_channel_ctx_t *ctx,
                             safety_channel_output_t *out) {
    out->axes_count = ctx->axes_count;
    out->sensor_ok = 1;

    for (uint8_t i = 0; i < ctx->axes_count; i++) {
        uint32_t mv = 0;
        drv_status_t ret = ctx->adc->read_mv(ctx->channel_indices[i], &mv);

        // 独立的开路/短路检测（不复用 fault_detector.c）
        if (ret != DRV_OK || mv < SAFETY_MV_LOW || mv > SAFETY_MV_HIGH) {
            out->sensor_ok = 0;
            out->axes[i] = 0;
            continue;
        }

        // 独立的简化校准（不复用 calibrator.c）
        // 不需要滤波/死区——仅用于比对，精度要求低于 Level 1
        uint16_t raw_approx = (uint16_t)(mv * 4095u / 3300u);
        out->axes[i] = (int32_t)raw_approx - (int32_t)ctx->center[i];
    }
}
```

### 12.5 Level 2 — 比较器

```c
// safety/safety_comparator.h
#pragma once
#include <stdint.h>
#include "safety_channel.h"

typedef enum {
    COMPARE_OK,
    COMPARE_DEGRADED,     // 超差但未超阈值
    COMPARE_FAIL,         // 持续超差 → 安全停止
} compare_result_t;

typedef struct {
    int32_t  tolerance[MAX_AXES];    // 允许的 L1-L2 偏差（每轴独立）
    uint8_t  fault_count[MAX_AXES];  // 连续超差计数
    uint8_t  fault_threshold;        // 允许的连续超差次数
} safety_comparator_t;

compare_result_t safety_compare(
    safety_comparator_t *cmp,
    const int32_t *l1_axes,
    const safety_channel_output_t *l2,
    uint8_t axes_count);
```

```c
// safety/safety_comparator.c
#include "safety_comparator.h"

compare_result_t safety_compare(
    safety_comparator_t *cmp,
    const int32_t *l1_axes,
    const safety_channel_output_t *l2,
    uint8_t axes_count)
{
    if (!l2->sensor_ok) {
        return COMPARE_FAIL;
    }

    compare_result_t worst = COMPARE_OK;

    for (uint8_t i = 0; i < axes_count; i++) {
        int32_t diff = l1_axes[i] - l2->axes[i];
        if (diff < 0) { diff = -diff; }

        if (diff > cmp->tolerance[i]) {
            cmp->fault_count[i]++;
            if (cmp->fault_count[i] >= cmp->fault_threshold) {
                worst = COMPARE_FAIL;
            } else if (worst < COMPARE_DEGRADED) {
                worst = COMPARE_DEGRADED;
            }
        } else {
            cmp->fault_count[i] = 0;
        }
    }
    return worst;
}
```

### 12.6 Level 3 — 条件喂狗 + 程序流监控

v3 的 `watchdog_feed()` 是**无条件喂狗**——只要 safety_task 在运行就喂。这不满足 SIL 2 要求。v4 改为条件喂狗：

```c
// safety/safety_watchdog.h
#pragma once
#include <stdint.h>
#include <stdbool.h>

// alive_token: 仅当 Level 2 比较通过时才更新
// 喂狗条件：token 正确 + 程序流标记完整
typedef struct {
    uint32_t expected_token;
    uint32_t actual_token;
    uint32_t cf_expected;       // 程序流期望签名
    uint32_t cf_actual;         // 程序流实际签名
} safety_wdg_ctx_t;

void safety_wdg_init(safety_wdg_ctx_t *ctx);
void safety_wdg_update_token(safety_wdg_ctx_t *ctx);   // L2 比较通过时调用
void safety_wdg_service(safety_wdg_ctx_t *ctx);         // 条件喂狗
```

```c
// safety/safety_watchdog.c
#include "safety_watchdog.h"

// 简单伪随机序列，防止代码意外跳转到 update_token 就通过
static uint32_t next_token(uint32_t seed) {
    return seed * 1103515245u + 12345u;
}

void safety_wdg_init(safety_wdg_ctx_t *ctx) {
    ctx->expected_token = 0xA5A5A5A5u;
    ctx->actual_token   = 0;
    ctx->cf_expected    = CF_MAIN_EXPECTED_SIGNATURE;
    ctx->cf_actual      = CF_SEED;
}

void safety_wdg_update_token(safety_wdg_ctx_t *ctx) {
    ctx->actual_token = ctx->expected_token;
    ctx->expected_token = next_token(ctx->expected_token);
}

void safety_wdg_service(safety_wdg_ctx_t *ctx) {
    bool token_ok = (ctx->actual_token != 0) &&
                    (next_token(ctx->actual_token) == ctx->expected_token);
    bool cf_ok    = (ctx->cf_actual == ctx->cf_expected);

    if (token_ok && cf_ok) {
        IWDG_feed();            // 内部看门狗
        ext_wdg_feed();         // 外部看门狗（如 MAX6369）
    }
    // else: 不喂狗 → 看门狗超时 → 硬件复位

    // 重置程序流标记，等待下一轮
    ctx->cf_actual = CF_SEED;
}
```

### 12.7 Level 3 — 独立硬件关断路径

**最后防线——不经过 RTOS、event_bus、状态机、驱动抽象层：**

```c
// safety/safe_shutdown.h
#pragma once
#include "stm32g4xx.h"   // 或 S32K144.h，直接访问寄存器

// 硬件直接关断：GPIO → 安全继电器 / STO (Safe Torque Off)
static inline void safe_shutdown_activate(void) {
    // 直接操作 GPIO 寄存器，不通过 gpio_writer_t 抽象层
    // 避免驱动层 bug 导致关断失效
    GPIOB->BRR = GPIO_PIN_ENABLE_OUT_Msk;  // 拉低使能输出

    // 设置安全状态标志（atomic）
    __atomic_store_n(&g_safety_shutdown_active, 1, __ATOMIC_RELEASE);

    // 尽力通过 CAN 发送紧急帧（不阻塞，不等待）
    // 即使 CAN 发送失败，硬件关断已经生效
}
```

### 12.10 附加诊断 — 平台认证安全库集成（DC ≥ 90%）

底层硬件诊断**不自研**，使用平台厂商提供的已认证安全库：

| 平台 | 安全库 | 认证等级 | 覆盖项 |
|------|--------|---------|--------|
| STM32G4 | **X-CUBE-STL** (ST 官方) | IEC 61508 SIL 2/3 | CPU/RAM/Flash/Clock 自检 |
| S32K144 | **SafeAssure Safety SW** (NXP) | IEC 61508 SIL 2, ISO 26262 ASIL B | CST(CPU)/ECC(RAM)/CRC(Flash)/CMU(Clock) |

> 使用预认证库的优势：减少 80%+ 的认证文档工作量，厂商提供 FMEDA 和安全手册，审核时直接引用。

#### 12.10.1 安全自检抽象接口

上层不直接调用 STL/NXP API，通过统一接口抽象，保持平台无关性：

```c
// safety/safety_stl.h — 安全自检接口（上层只依赖这个）
#pragma once
#include <stdint.h>
#include <stdbool.h>

typedef enum {
    STL_RESULT_PASS,
    STL_RESULT_FAIL,
    STL_RESULT_IN_PROGRESS,
} stl_result_t;

typedef struct {
    stl_result_t (*cpu_test_partial)(void);     // CPU 指令集测试（分片执行）
    stl_result_t (*ram_test_partial)(void);     // RAM March C- 测试（分片执行）
    stl_result_t (*flash_crc_check)(void);      // Flash CRC 完整性
    stl_result_t (*clock_cross_check)(void);    // 时钟交叉校验
    void         (*stack_overflow_init)(void);  // MPU/Sentinel 初始化
} safety_stl_t;

// 由平台层注册
void platform_register_stl(const safety_stl_t *stl);
const safety_stl_t* platform_get_stl(void);
```

#### 12.10.2 STM32G4 实现（X-CUBE-STL）

```c
// platform/stm32g4/stl_stm32g4.c
#include "safety_stl.h"
#include "stm32_stl_api.h"   // X-CUBE-STL 官方头文件

static stl_result_t stm32_cpu_test(void) {
    STL_Status_t res = STL_SCH_RunCpuTMx();
    return (res == STL_PASSED) ? STL_RESULT_PASS : STL_RESULT_FAIL;
}

static stl_result_t stm32_ram_test(void) {
    STL_Status_t res = STL_SCH_RunRamTMx();
    switch (res) {
        case STL_PASSED:      return STL_RESULT_PASS;
        case STL_PARTIAL_PASS: return STL_RESULT_IN_PROGRESS;
        default:              return STL_RESULT_FAIL;
    }
}

static stl_result_t stm32_flash_crc(void) {
    STL_Status_t res = STL_SCH_RunFlashTMx();
    return (res == STL_PASSED) ? STL_RESULT_PASS : STL_RESULT_FAIL;
}

static stl_result_t stm32_clock_check(void) {
    STL_Status_t res = STL_SCH_RunClockTMx();
    return (res == STL_PASSED) ? STL_RESULT_PASS : STL_RESULT_FAIL;
}

static void stm32_stack_init(void) {
    // MPU 配置在 CubeMX 中生成，此处激活 Stack Sentinel 区域
    HAL_MPU_ConfigRegion(&mpu_stack_sentinel_config);
}

const safety_stl_t stl_stm32g4 = {
    .cpu_test_partial  = stm32_cpu_test,
    .ram_test_partial  = stm32_ram_test,
    .flash_crc_check   = stm32_flash_crc,
    .clock_cross_check = stm32_clock_check,
    .stack_overflow_init = stm32_stack_init,
};
```

#### 12.10.3 S32K144 实现（NXP Safety SW）

```c
// platform/s32k144/stl_s32k144.c
#include "safety_stl.h"
#include "safety_test.h"      // NXP SafeAssure API

static stl_result_t s32k_cpu_test(void) {
    // S32K144 Core Self-Test (CST)
    uint32_t result = FS_CM4_CST_CPU();
    return (result == FS_PASS) ? STL_RESULT_PASS : STL_RESULT_FAIL;
}

static stl_result_t s32k_ram_test(void) {
    // S32K144 ECC 由 EIM/ERM 硬件模块持续监控
    // 此处检查 ERM 错误寄存器
    if (ERM->SR0 & ERM_SR0_NCE0_MASK) {
        ERM->SR0 = ERM_SR0_NCE0_MASK;  // 清除标志
        return STL_RESULT_FAIL;          // 不可纠正错误
    }
    return STL_RESULT_PASS;
}

static stl_result_t s32k_flash_crc(void) {
    uint32_t result = FS_CM4_FLASH_CRC(flash_start, flash_end, expected_crc);
    return (result == FS_PASS) ? STL_RESULT_PASS : STL_RESULT_FAIL;
}

static stl_result_t s32k_clock_check(void) {
    // S32K144 CMU (Clock Monitor Unit) 硬件自动监控
    // 此处检查 CMU 状态寄存器
    if (CMU_FC_0->SR & CMU_FC_SR_FLL_MASK) {
        return STL_RESULT_FAIL;   // 频率丢失
    }
    return STL_RESULT_PASS;
}

const safety_stl_t stl_s32k144 = {
    .cpu_test_partial  = s32k_cpu_test,
    .ram_test_partial  = s32k_ram_test,
    .flash_crc_check   = s32k_flash_crc,
    .clock_cross_check = s32k_clock_check,
    .stack_overflow_init = s32k_stack_init,
};
```

#### 12.10.4 safety_task 中调用（替代自研诊断）

```c
// safety_task 中的附加诊断调用（替代 v4 早期的自研实现）
static void safety_diagnostics(void) {
    const safety_stl_t *stl = platform_get_stl();

    // CPU 自检（分片执行，每个 safety_task 周期执行一片）
    if (stl->cpu_test_partial() == STL_RESULT_FAIL) {
        safe_shutdown_activate();
    }

    // RAM 测试（分片 March C-，多个周期完成一轮全覆盖）
    if (stl->ram_test_partial() == STL_RESULT_FAIL) {
        safe_shutdown_activate();
    }

    // 时钟交叉校验
    if (stl->clock_cross_check() == STL_RESULT_FAIL) {
        safe_shutdown_activate();
    }

    // Flash CRC（低频执行，每 N 个周期一次）
    static uint16_t flash_check_counter = 0;
    if (++flash_check_counter >= 500) {  // 每 1 秒
        flash_check_counter = 0;
        if (stl->flash_crc_check() == STL_RESULT_FAIL) {
            safe_shutdown_activate();
        }
    }
}
```

#### 12.10.5 安全变量互补存储（应用层，仍需自研）

认证库不覆盖应用层的安全关键变量保护，此部分仍需自研：

```c
// safety/safe_var.h — 安全关键变量互补存储
#pragma once
#include <stdint.h>
#include <stdbool.h>

typedef struct {
    int32_t  value;
    int32_t  complement;   // 应等于 ~value
} safe_var_t;

static inline bool safe_var_check(const safe_var_t *v) {
    return v->value == (int32_t)(~(uint32_t)v->complement);
}

static inline void safe_var_write(safe_var_t *v, int32_t val) {
    v->value      = val;
    v->complement = (int32_t)(~(uint32_t)val);
}
```

### 12.11 自研 vs 认证库职责划分

```
┌──────────────────────────────────────────────────────────────┐
│                   安全功能职责划分                              │
├──────────────────────────┬───────────────────────────────────┤
│  使用认证库（平台提供）     │  自研（应用层安全逻辑）             │
├──────────────────────────┼───────────────────────────────────┤
│  CPU 指令集自检            │  L2 独立传感器采集+计算            │
│  RAM March C- 测试        │  L1/L2 输出比较器                 │
│  Flash CRC 完整性          │  条件喂狗策略（token + 程序流）    │
│  时钟交叉校验              │  程序流监控（checkpoint 签名链）   │
│  MPU 内存分区              │  独立硬件关断路径                 │
│  ECC 硬件纠错(S32K)       │  安全变量互补存储                 │
│  WDG 硬件驱动             │  安全状态管理                     │
├──────────────────────────┴───────────────────────────────────┤
│  原则：硬件诊断用认证库，应用安全逻辑自研                         │
│  认证时：认证库直接引用厂商安全手册，自研部分提交 V&V 报告         │
└──────────────────────────────────────────────────────────────┘
```

### 12.12 单 MCU 满足 SIL 2 的必要措施

| 措施 | 目的 | 实现方式 |
|------|------|---------|
| CPU/RAM/Flash/Clock 自检 | 硬件故障诊断 | **X-CUBE-STL / NXP Safety SW**（已认证） |
| MPU 内存分区 | L1/L2 内存隔离 | CubeMX/SDK 配置 |
| 独立代码实现 | 软件多样性 | L2 信号计算不复用 L1 `axis_pipeline` 代码 |
| 外部看门狗 | 独立于 CPU | 外挂 WDG IC（如 MAX6369） |
| 独立关断路径 | 硬件直达 | GPIO 直驱安全继电器，不经软件抽象层 |
| 程序流监控 | 检测跳转错误 | checkpoint 签名链（自研） |
| 互补存储 | 应用层安全变量 | `safe_var_t`（自研） |

> **SIL 3 / PL e** 才强制要求独立 MCU（双处理器）或锁步核（如 S32K3 Cortex-M7 lockstep）。

## 十六、设计原则对照总结

### SOLID 原则合规性

| 原则 | 原设计状态 | 改进方案 | 改进后状态 |
|------|-----------|---------|-----------|
| **S** — 单一职责 | `joystick_core.c` 五职责合一；`main_app.c` 混合组装与运行；协议引擎兼做值转换+打包 | 信号管线拆分；组装与运行分离；v4 新增 output_adapter 将值转换从协议引擎剥离 | 每个模块只有一个变更理由 |
| **O** — 开闭原则 | `product_config.h` 的 `#elif` 链；CMake 中硬编码安全型号名 | 每产品独立 `.c`；v4 改为 `SAFETY_ENABLED` 参数驱动 | 新增产品/安全型号不修改任何已有代码 |
| **L** — 里氏替换 | 协议接口强制所有方法，部分协议需写空函数 | 核心接口 + 能力位掩码 + 可选扩展方法 | 所有 CAN 协议都可以安全替换使用 |
| **I** — 接口隔离 | `adc_driver_t` 包含所有方法 | 拆分为 `adc_reader` / `adc_voltage_reader` / `adc_continuous` | 每个模块只依赖自己需要的最小接口 |
| **D** — 依赖倒置 | `#if defined(PLATFORM_STM32G4)` 硬编码平台 | 平台注册表 + 显式依赖注入 | 上层完全不知道下层具体实现 |

### 设计模式使用

| 模式 | 用途 | 位置 |
|------|------|------|
| **观察者模式** | 模块间松耦合事件通信 | `event/event_bus.h` |
| **策略模式** | 可插拔的死区/校准算法 | `deadzone_fn_t` / `calibrate_fn_t` |
| **状态模式** | 设备生命周期管理 | `app/device_state.h` |
| **扫描-缓冲模式** | 硬件扫描与数据消费解耦 | `input/scan_buffer.h` + `board/board_scan.h` |
| **Adapter 模式** | v4: 管线内部值 → 自描述输出 | `output/output_adapter.h` |
| **Anti-Corruption Layer** | v4: 协议层不解释管线内部表示 | `output_data_t` 自描述边界对象 |
| **Tagged Union** | v4: 编译期类型安全的协议配置 | `protocol_bundle_t` |
| **双通道比较** | v4: E-Gas L2 独立计算+比对 | `safety/safety_comparator.h` |

### v4 相比 v3 的关键改进

| 改进点 | v3 设计 | v4 改进 | 原因 |
|--------|---------|---------|------|
| 信号链接口 | 泛型 chain（`int16_t`）+ 显式 pipeline（`int32_t`）并存 | 删除泛型 chain，统一为显式 `axis_pipeline`（`uint16_t→int32_t` 分界） | 消除类型不匹配和概念混淆 |
| 值域转换 | 嵌入协议引擎内部 | 独立 `output_adapter` 层（纯函数） | SRP + 可复用（HMI/日志/调试均可调用） |
| 协议引擎输入 | `joystick_data_t`（非自描述） | `output_data_t`（自描述，含值范围） | Anti-Corruption Layer |
| max_deflection | 单一值（假设所有轴同分辨率） | per-axis（`uint16_t max_deflection[MAX_AXES]`） | 支持 JS-300 混合 12-bit/14-bit 传感器 |
| 产品描述符 | 缺少 protocol 字段 | 新增 `protocol_bundle` + `output_maps` + `safety_config` | 完整的产品自描述 |
| 安全架构 | safety_task 旁观式监控（读 L1 数据判断异常） | E-Gas 三层：L2 独立采集+比对，L3 条件喂狗+硬件关断 | SIL 2 / PL d 合规 |
| 看门狗 | 无条件喂狗 | 条件喂狗（L2 token + 程序流标记双重校验） | IEC 61508-7 要求 |
| 安全关断 | 依赖 event_bus → 状态机（共因失效） | GPIO 直驱安全继电器（不经 RTOS/event_bus） | 独立关断路径 |
| CMake 安全模块 | 硬编码 `JS300 OR JS400` | `SAFETY_ENABLED` 参数驱动 | OCP：新增安全型号不改 CMakeLists.txt |

### 关键约束

所有改进都保持了嵌入式 C 的实用性与功能安全合规性：
- 纯 C99 标准，无 GCC/编译器特有扩展（`_Atomic` 来自 C11，安全模块需指定 `-std=c11`）
- 无虚函数表 / RTTI
- 无动态内存分配（`malloc`/`free`）
- 无 C++ 异常
- 所有缓冲区静态分配
- 事件总线订阅者数量编译时确定
- 所有初始化顺序显式可控（无 constructor / 隐式初始化）
- MISRA C:2012 可达标
- IEC 61508 SIL 2 / ISO 13849 PL d 架构合规（E-Gas 三层 + 独立关断路径）
