# 工业摇杆产品线软件架构设计

## 一、问题分析

一个工业摇杆产品线通常面临三个维度的变化：

```
维度1：产品型号（硬件差异）
├── JS-100：单轴摇杆，2按键，CAN输出
├── JS-200：双轴摇杆，4按键，CAN输出
├── JS-300：三轴摇杆+旋钮，8按键，CAN+安全功能
└── JS-400：双轴+力反馈，12按键，CAN+安全功能

维度2：通信协议（客户需求差异）— 仅 CAN 总线族
├── CAN 2.0B（自定义协议）
├── CANopen（DS401 / DS406）
├── SAE J1939
└── CANopen Safety（EN 50325-5）

维度3：运行平台（芯片差异）
├── STM32F103
├── STM32G431
├── S32K144
└── 未来平台
```

如果用排列组合，4×4×3 = 48 种配置。传统做法是每种配置一个工程，维护噩梦。
正确的做法是用**组合而非继承**，通过架构设计让这三个维度正交独立。

### 架构演进历程

| 变化点 | v1 设计 | v2 改进 | v3 改进 | v4 改进 | v5 改进 | 对应原则 |
|--------|---------|---------|---------|---------|---------|----------|
| 摇杆核心 | 单函数五职责 | 信号处理链 pipeline | 同 v2 | 删除泛型 chain，统一为显式 `axis_pipeline`（`uint16_t→int32_t` 类型分界） | 同 v4 | SRP |
| 输入采集 | 直接调 ADC | 同 v1 | `scan_buffer_t` + `board_scanner_t` 扫描-缓冲解耦 | 同 v3 | 三层模型：sensor backend → `input_acq_t` → `scan_buffer_t`；DMA 双缓冲+序号+屏障 | SRP+OCP |
| 按键采集 | 仅 GPIO | 同 v1 | `buf_btn_source`（MUX 通道阈值检测，支持多键同按） | 同 v3 | 同 v3（消费端不变，生产端改为 `input_acq_t`） | ISP |
| 产品配置 | `#elif` 链 | 每产品独立 `.c` | 同 v2（满量程由 `resolution_bits` 推导） | 新增 `protocol_bundle` 字段 + per-axis `max_deflection` | 同 v4 | OCP |
| 协议接口 | 7方法全实现 | 能力位掩码 | 同 v2 | 协议引擎只管打包，值域转换移至 `output_adapter` | 同 v4 | SRP+Adapter |
| 协议输出 | 值转换嵌入引擎 | 同 v1 | Profile 驱动 | 新增 `output_adapter` 层，协议引擎接收自描述 `output_data_t` | 同 v4 | Anti-Corruption |
| ADC 接口 | 胖接口 | ISP 拆分 | 新增 SPI 接口 | 同 v3 | `adc_continuous_t` 增加完成回调，纳入 `frame_publisher_t` 双缓冲 | ISP |
| CAN 接口 | — | — | `can_port_t` 过渡设计 | 同 v3 | `can_port_t`（`try_recv`/`wait_recv`，callback 仅保留在平台层内部） | DIP |
| 平台注册 | `#if defined` | `__attribute__((constructor))` | 显式 `platform_init()`（MISRA 合规） | 同 v3 | 新增 `async_*` 原语注册 + `event_queue_backend` 注册 | DIP |
| 异步原语 | 无 | 无 | 无 | 无 | `async_notify_t` / `async_queue_t` / `async_barrier`（仅覆盖 ISR→task 边界） | ISP |
| 归一化范围 | ±10000 固定 | 同 v1 | 内部无符号原始范围，协议层按需转换 | 不归一化；`output_adapter` 在协议打包前一次性缩放 | 同 v4 | — |
| 任务架构 | 裸循环 | 裸循环 + `os_delay` | RTOS 多任务（CMSIS-RTOS v2） | safety_task 升级为 L2 独立计算通道 | main_task 用 `input_acq_t`，proto_task 用 `can_port_t.wait_recv` | SRP+安全 |
| 事件总线 | 无 | 同步分发 | 增加 ISR 安全延迟分发 | 同 v3 | 控制面/数据面分离 + `event_queue_backend_t` 注入（摆脱 ARM IPSR 依赖） | 观察者 |
| 协议输出 | CAN + Analog 混合 | 同 v1 | 仅 CAN 族（CANopen/J1939/CAN_Raw） | 同 v3 | 同 v3 | 精简 |
| 死区策略 | 硬编码线性死区 | 可插拔策略（线性/S曲线/指数） | 同 v3 | 新增 | 同 v4 |
| 功能安全 | 无 | 无 | safety_task 旁观式监控 | E-Gas 三层架构（L1功能/L2独立监控/L3条件喂狗+硬件关断） | 同 v4 | IEC 61508 SIL 2 |
| 安全关断 | 无 | 无 | 依赖 event_bus → 状态机 | 独立硬件关断路径（GPIO 直驱安全继电器，不经 RTOS） | 同 v4 | PL d Cat.3 |
| 看门狗 | 无 | 无 | 无条件喂狗 | 条件喂狗（L2 比对通过 + 程序流标记正确时才喂） | 同 v4 | IEC 61508-7 |

## 三、驱动接口层设计（ISP 改进：按使用场景拆分）

### 3.1 ADC 接口 — 按使用场景隔离

原设计将所有 ADC 功能放在一个 `adc_driver_t` 中。摇杆核心只需 `read()`，安全监控需要 `read_mv()`，DMA 采集才需要 `start_continuous()`。按接口隔离原则拆分为三个独立接口：

```c
// drivers/adc_reader.h — 基础读取接口，大多数模块只需要这个
#pragma once
#include "drv_common.h"

typedef struct {
    drv_status_t (*init)(void);
    drv_status_t (*read)(adc_channel_id_t ch, uint16_t *raw);
    drv_status_t (*deinit)(void);
} adc_reader_t;
```

```c
// drivers/adc_voltage_reader.h — 电压读取接口，安全监控/诊断需要
#pragma once
#include "drv_common.h"

typedef struct {
    drv_status_t (*read_mv)(adc_channel_id_t ch, uint32_t *mv);
} adc_voltage_reader_t;
```

```c
// drivers/adc_continuous.h — DMA 连续采集接口，高速采集模块需要
#pragma once
#include "drv_common.h"

typedef struct {
    drv_status_t (*start_continuous)(adc_channel_id_t *chs, uint8_t cnt,
                                     uint16_t *buf, uint16_t buf_len);
    drv_status_t (*stop_continuous)(void);
} adc_continuous_t;
```

```c
// drivers/drv_common.h — 公共类型定义
#pragma once
#include <stdint.h>

typedef enum {
    DRV_OK = 0,
    DRV_ERROR,
    DRV_TIMEOUT,
    DRV_BUSY,
    DRV_INVALID_ARG,
    DRV_NOT_SUPPORTED,
} drv_status_t;

typedef enum {
    ADC_CH_AXIS_X,
    ADC_CH_AXIS_Y,
    ADC_CH_AXIS_Z,
    ADC_CH_KNOB,
    ADC_CH_VREF,
    ADC_CH_TEMP,
    ADC_CH_SUPPLY,
    ADC_CH_MAX
} adc_channel_id_t;
```

平台实现可以同时满足多个接口：

```c
// platform/stm32g4/adc_stm32g4.c
const adc_reader_t          adc_reader_stm32g4 = { .init = ..., .read = ..., .deinit = ... };
const adc_voltage_reader_t  adc_voltage_stm32g4 = { .read_mv = ... };
const adc_continuous_t      adc_continuous_stm32g4 = { .start_continuous = ..., .stop_continuous = ... };
```

各模块只依赖自己需要的最小接口：

```c
// 摇杆核心只依赖基础读取
void joystick_core_init(joystick_core_t *js,
                         const adc_reader_t *adc,
                         const gpio_reader_t *gpio);

// 安全监控依赖电压读取接口
void safety_monitor_init(safety_monitor_t *sm,
                          const adc_voltage_reader_t *adc,
                          const gpio_reader_t *gpio);
```

### 3.2 CAN 端口接口（v5 改进：callback 内收到平台层）

v4 原设计通过 `register_rx_cb(can_rx_callback_t cb)` 将 ISR 回调暴露给上层。这导致：
- 上层必须在回调中处理 ISR 语义（重入约束、栈大小限制）
- STM32 弱符号回调、NXP 中断分发句柄、PC 线程的差异泄漏到应用层
- 不同平台的回调注册方式完全不同

v5 改为 **端口模型**：callback 只存在于平台适配层内部，上层通过 `try_recv` / `wait_recv` 消费帧。

```c
// drivers/drv_can.h
#pragma once
#include "drv_common.h"

typedef struct {
    uint32_t id;
    uint8_t  data[64];   // 支持 CAN-FD
    uint8_t  dlc;
    uint8_t  is_extended; // 0=标准帧 1=扩展帧
    uint8_t  is_fd;       // 0=CAN2.0 1=CAN-FD
} can_frame_t;

typedef struct {
    drv_status_t (*init)(void *ctx, uint32_t baudrate);

    // 发送（任务上下文调用）
    drv_status_t (*send)(void *ctx, const can_frame_t *frame);

    // 非阻塞接收：立即返回
    // DRV_OK=取到帧，DRV_BUSY=队列为空
    drv_status_t (*try_recv)(void *ctx, can_frame_t *frame);

    // 阻塞接收：让出 CPU 等待帧到达
    // RTOS 平台 → osMessageQueueGet(timeout)
    // 裸机平台 → ring buffer + WFI
    // PC mock  → 条件变量
    drv_status_t (*wait_recv)(void *ctx, can_frame_t *frame, uint32_t timeout_ms);

    drv_status_t (*set_filter)(void *ctx, uint32_t id, uint32_t mask);
    drv_status_t (*get_bus_state)(void *ctx, uint8_t *tx_err, uint8_t *rx_err);
    drv_status_t (*deinit)(void *ctx);

    void *ctx;  // 平台上下文（含内部 ring buffer / async_queue 等）
} can_port_t;
```

平台适配层内部结构示例：

```c
// platform/stm32g4/can_stm32g4.c（内部，不暴露给上层）
#include "async_queue.h"

typedef struct {
    async_queue_t rx_queue;           // ISR → 任务 的帧队列
    can_frame_t   rx_buf[16];         // 队列存储
    // ... HAL 句柄等平台细节
} can_stm32g4_ctx_t;

// STM32 HAL CAN RX 中断回调（平台内部，不暴露）
void HAL_CAN_RxFifo0MsgPendingCallback(CAN_HandleTypeDef *hcan) {
    can_frame_t frame;
    // ... 从 HAL 读帧 ...
    async_queue_put_from_isr(&g_can_ctx.rx_queue, &frame);  // ISR 安全入队
}

// 对外接口实现
static drv_status_t stm32_try_recv(void *ctx, can_frame_t *frame) {
    can_stm32g4_ctx_t *c = (can_stm32g4_ctx_t *)ctx;
    return async_queue_get(&c->rx_queue, frame, 0) ? DRV_OK : DRV_BUSY;
}

static drv_status_t stm32_wait_recv(void *ctx, can_frame_t *frame, uint32_t timeout_ms) {
    can_stm32g4_ctx_t *c = (can_stm32g4_ctx_t *)ctx;
    return async_queue_get(&c->rx_queue, frame, timeout_ms) ? DRV_OK : DRV_TIMEOUT;
}
```

> **核心原则**：callback 只存在于平台层内部。上层代码永远不接触 ISR callback，只通过 `try_recv` / `wait_recv` 消费帧。

### 3.4 平台注册机制（DIP 改进：依赖方向完全反转）

原设计中 `app_config.c` 通过 `#if defined(PLATFORM_STM32G4)` 直接引用平台符号，编译时依赖方向从上往下。改进后，平台层主动向上注册，上层代码完全不知道具体平台的存在。

```
改进前（依赖方向向下）：
  app_config.c ──→ platform/stm32g4/adc_stm32g4.c

改进后（依赖方向向上）：
  app_config.c ──→ drivers/adc_reader.h ←── platform/stm32g4/adc_stm32g4.c
```

```c
// platform/platform_registry.h — 上层只依赖这个抽象接口
#pragma once
#include "adc_reader.h"
#include "adc_voltage_reader.h"
#include "drv_can.h"
#include "drv_gpio.h"
#include "drv_spi.h"

void platform_register_adc(const adc_reader_t *drv);
void platform_register_adc_voltage(const adc_voltage_reader_t *drv);
void platform_register_can(const can_port_t *port);
void platform_register_gpio_reader(const gpio_reader_t *drv);
void platform_register_gpio_writer(const gpio_writer_t *drv);
void platform_register_spi(const spi_driver_t *drv);
void platform_register_stl(const safety_stl_t *stl);  // v4: 安全自检库注册

const adc_reader_t*          platform_get_adc(void);
const adc_voltage_reader_t*  platform_get_adc_voltage(void);
const can_port_t*            platform_get_can(void);
const gpio_reader_t*         platform_get_gpio_reader(void);
const gpio_writer_t*         platform_get_gpio_writer(void);
const spi_driver_t*          platform_get_spi(void);
const safety_stl_t*          platform_get_stl(void);   // v4

// 显式初始化入口 — 每个平台实现一个
// 替代 __attribute__((constructor))，满足 MISRA C Rule 1.2
void platform_init(void);
```

```c
// platform/stm32g4/platform_init.c
// 显式调用，初始化顺序确定性可控，可设断点调试
// 不依赖任何 GCC 扩展，满足功能安全要求

#include "platform_registry.h"
#include "adc_stm32g4.h"
#include "can_stm32g4.h"
#include "gpio_stm32g4.h"
#include "spi_stm32g4.h"

void platform_init(void) {
    platform_register_adc(&adc_reader_stm32g4);
    platform_register_adc_voltage(&adc_voltage_stm32g4);
    platform_register_can(&can_port_stm32g4);
    platform_register_gpio_reader(&gpio_reader_stm32g4);
    platform_register_gpio_writer(&gpio_writer_stm32g4);
    platform_register_spi(&spi_driver_stm32g4);
    #if SAFETY_ENABLED
    platform_register_stl(&stl_stm32g4);   // v4: 注册 X-CUBE-STL 认证安全库
    #endif
}
```

> **为什么不用 `__attribute__((constructor))`？**
>
> | 对比项 | `__attribute__((constructor))` | 显式 `platform_init()` |
> |--------|-------------------------------|----------------------|
> | MISRA C 合规 | 违反 Rule 1.2（实现定义行为） | 合规 |
> | 初始化顺序 | 不确定，跨编译单元无法保证 | 代码中显式控制 |
> | 可调试性 | 在 `main()` 之前执行，断点难设 | 普通函数调用，随时断点 |
> | 功能安全认证 | IEC 61508 / ISO 13849 审核不通过 | 可审核、可追溯 |
> | 可移植性 | GCC 专属，IAR/ARMCC/Green Hills 不支持 | 纯 C 标准，所有编译器通用 |

```c
// main.c — 启动顺序清晰（使用 CMSIS-RTOS v2 API）
#include "cmsis_os2.h"

int main(void) {
    // 1. 硬件初始化（BSP / CubeMX 生成）
    HAL_Init();
    SystemClock_Config();

    // 2. 平台驱动注册（显式调用，替代 constructor）
    platform_init();

    // 3. RTOS 内核初始化
    osKernelInitialize();

    // 4. 创建应用任务
    app_tasks_create();

    // 5. 启动调度器（不返回）
    osKernelStart();
}
```

链接器决定哪个平台的 `platform_init.c` 参与编译，上层代码零修改。每个平台只需提供自己的 `platform_init()` 实现即可。

### 3.7 异步原语层（v5 新增：ISR 边界解耦）

#### 3.7.1 设计动机

架构中存在三类异步交互模式，它们的平台差异大于 RTOS 差异：

| 场景 | ISR 侧行为 | 任务侧行为 | 平台差异 |
|------|-----------|-----------|---------|
| ADC DMA 完成 | 更新双缓冲索引 + 置位通知 | 取最新完成帧 | STM32 HAL callback vs NXP eDMA callback vs PC 定时器 |
| CAN RX | 帧入环形缓冲 + 唤醒消费者 | 阻塞/轮询取帧 | STM32 弱符号回调 vs NXP 中断分发句柄 vs PC 线程 |
| 事件总线 ISR 发布 | 事件入队列 | 延迟分发 | CMSIS-RTOS 依赖 ARM IPSR 寄存器自动检测，非 ARM 平台不可用 |

**CMSIS-RTOS v2 是 OS 抽象，不是 I/O 语义抽象。** 它解决了任务调度/互斥/延时的可移植性，但不解决 ISR→任务数据传递的语义统一。需要一层极薄的异步原语，专门覆盖 **ISR→任务边界**。

#### 3.7.2 职责边界（硬约束）

```
async_* 只解决 "ISR → task" 边界，不处理 task → task 同步。

┌───────────────────────────────────┐
│        async primitives           │  ← 本层
│  async_notify_t: ISR 唤醒任务     │
│  async_queue_t:  ISR 投递数据     │
│  async_barrier:  内存可见性屏障    │
└───────────────────────────────────┘
          ↑ 不替代 ↓
┌───────────────────────────────────┐
│        CMSIS-RTOS v2              │  ← OS 层
│  任务调度 / mutex / semaphore     │
│  timer / 绝对延时                 │
└───────────────────────────────────┘
```

规则：
- 任务调度、任务间互斥、定时器 → 继续用 CMSIS-RTOS v2
- ISR 投递数据/唤醒消费者 → 使用 `async_notify_t` / `async_queue_t`
- 内存可见性保证 → 使用 `async_write_barrier()` / `async_read_barrier()`

#### 3.7.3 内存屏障原语

`volatile` 只约束编译器重排序，不足以表达跨核/总线/外设 DMA 的可见性。双缓冲模式中必须显式插入屏障。

```c
// async/async_barrier.h — 平台无关的内存屏障抽象
#pragma once

// 写屏障：保证之前的写操作对其他观察者可见后，才执行后续写操作
// 用途：生产者在更新双缓冲数据后、发布 ready_idx 之前调用
void async_write_barrier(void);

// 读屏障：保证之前的读操作完成后，才执行后续读操作
// 用途：消费者读取 ready_idx 后、访问缓冲数据之前调用
void async_read_barrier(void);
```

各平台实现：

```c
// platform/stm32g4/async_barrier_stm32.c
#include "async_barrier.h"
void async_write_barrier(void) { __DMB(); }  // ARM Data Memory Barrier
void async_read_barrier(void)  { __DMB(); }

// platform/riscv/async_barrier_riscv.c（未来平台）
void async_write_barrier(void) { __asm volatile("fence w,w" ::: "memory"); }
void async_read_barrier(void)  { __asm volatile("fence r,r" ::: "memory"); }

// platform/test/async_barrier_mock.c
// PC 上 volatile + 编译器屏障即足够（单核模拟）
void async_write_barrier(void) { __asm volatile("" ::: "memory"); }
void async_read_barrier(void)  { __asm volatile("" ::: "memory"); }
```

#### 3.7.4 通知原语

ISR 侧发信号唤醒任务，任务侧阻塞等待或轮询。


#### 3.7.5 队列原语

ISR 侧投递定长数据，任务侧阻塞或非阻塞取出。

```c
// async/async_queue.h — ISR → 任务 的单向数据队列

#include <stdbool.h>
```

#### 3.7.6 帧发布点封装

双缓冲 + 序号的发布/消费操作容易写错顺序。封装为专用函数，确保 barrier 顺序正确：

```c
// async/frame_publisher.h — 双缓冲帧发布/消费
#pragma once
#include "async_barrier.h"
#include "async_notify.h"
#include <stdint.h>

// 帧缓冲区描述符（生产者和消费者共享）
typedef struct {
    void         *buffers[2];       // 双缓冲区指针
    uint32_t      frame_size;       // 单帧字节数
    volatile uint8_t  ready_idx;    // 当前可读缓冲区索引
    volatile uint32_t seq;          // 单调递增序号
    async_notify_t    notify;       // 帧就绪通知
} frame_publisher_t;

// ISR/DMA 完成回调中调用：发布新帧
static inline void frame_publish(frame_publisher_t *fp, uint8_t completed_idx) {
    async_write_barrier();          // 保证缓冲区数据先可见
    fp->ready_idx = completed_idx;
    fp->seq++;
    async_notify_give_from_isr(&fp->notify);
}

// 任务上下文调用：消费最新帧
// 返回 DRV_OK=新帧, DRV_BUSY=无更新
static inline drv_status_t frame_consume(frame_publisher_t *fp,
                                          void *out,
                                          uint32_t *out_seq) {
    uint32_t s = fp->seq;
    uint8_t  idx = fp->ready_idx;
    async_read_barrier();           // 保证先读到索引再读数据
    memcpy(out, fp->buffers[idx], fp->frame_size);
    *out_seq = s;
    return DRV_OK;
}
```

## 四、输入采集层（v5 改进：三层输入模型 + 异步解耦）

### 4.1 设计动机

原设计假设"轴 = ADC 通道"，`joystick_update()` 直接调用 `adc->read(ch, &raw)`。但实际工业摇杆的输入源远比这复杂：

```
实际硬件拓扑（以 JS-300 为例）：
         5V ──┬── R1 ──┬── GND     5V ──┬── R1 ──┬── GND
              │ Button │                │ Button │
              2.5V     │                2.5V     │
                ↓      ↓                  ↓      ↓
        ┌──────────────────────────────────────────┐
        │       Analog MUX (CD74HC4067)            │
        │  CH0: Axis-X pot    CH4: Button-0 (2.5V) │
        │  CH1: Axis-Y pot    CH5: Button-1 (2.5V) │
        │  CH2: Axis-Z pot    CH6: Button-2 (2.5V) │
        │  CH3: (reserved)    CH7: Button-3 (2.5V) │
        └──────────────┬───────────────────────────┘
                       │ COM (共用输出)
                       ↓
                  ┌─────────┐
                  │   ADC   │ ← 同一个 ADC 通道
                  └─────────┘
```

关键约束：
1. **轴和按键共用 MUX+ADC 资源**——不能各自独立读取，必须协调扫描顺序
2. **按键采用逐通道 5V→2.5V 分压**——每个按键占独立 MUX 通道，按下时读 ≥2.5V，支持多键同按
3. **不同产品硬件差异大**——有的用 MUX，有的直接 ADC 多通道，有的用 DMA 连续采集
4. **拉模式与推模式混合**——MUX/直接 ADC 是同步拉取，DMA/SPI DMA 是异步推送

若将 MUX+ADC 作为整体资源封装进输入源，会导致轴源和按键源耦合在同一个扫描器上，缺乏平台化能力。而 v4 的 `board_scanner_t.scan()` 同步模型让 DMA 冒充同步扫描，掩盖了异步语义差异。因此 v5 将输入侧正式拆为三层。

### 4.2 三层输入模型（v5 核心改进）

v4 的 `board_scanner_t` 把"同步拓扑编排"和"异步采集机制"混在一个 `scan()` 接口中。v5 将其拆为三层：

```
  ┌─────────────────────────────────────────────────────────────┐
  │  Layer 1: Sensor Backend（平台层内部，不暴露给上层）           │
  │  负责 ADC/SPI/GPIO/MUX/DMA/HAL callback                     │
  │  ┌────────────┐ ┌──────────────┐ ┌──────────────┐           │
  │  │mux_adc_    │ │direct_adc_   │ │dma_adc_      │           │
  │  │backend     │ │backend       │ │backend       │           │
  │  │(同步拉取)   │ │(同步拉取)     │ │(异步推送)     │           │
  │  └─────┬──────┘ └──────┬───────┘ └──────┬───────┘           │
  │        │               │                │                    │
  │  ┌─────┴───────┐ ┌─────┴──────┐  ┌──────┴──────────────┐   │
  │  │adc_reader_t │ │adc_reader_t│  │adc_continuous_t     │   │
  │  │gpio_writer_t│ │            │  │frame_publisher_t    │   │
  │  └─────────────┘ └────────────┘  │async_notify_t       │   │
  │                                  └─────────────────────┘   │
  └──────────────────────────┬──────────────────────────────────┘
                             │ 统一接口
                             ▼
  ┌─────────────────────────────────────────────────────────────┐
  │  Layer 2: input_acq_t（采集后端接口，上层唯一入口）            │
  │  acquire_latest(ctx, &buf, &seq) → DRV_OK / DRV_BUSY       │
  │  对上提供"最新一致性快照 + 序号"                               │
  └──────────────────────────┬──────────────────────────────────┘
                             │ 写入
                             ▼
  ┌─────────────────────────────────────────────────────────────┐
  │  Layer 3: scan_buffer_t（通用样本载体，零依赖）               │
  │  被 buf_axis_source / buf_btn_source 消费                   │
  └─────────┬───────────────────┬───────────────────────────────┘
            │ 只读               │ 只读
            ↓                    ↓
  ┌──────────────────┐ ┌──────────────────────┐
  │  buf_axis_source │ │  buf_btn_source      │
  │  按 channel 取值 │ │  阈值检测 → bitmask  │
  └──────────────────┘ └──────────────────────┘
```

### 4.3 input_acq_t — 统一采集接口（v5 新增）

替代 v4 的 `board_scanner_t`。不再区分"扫描"和"取帧"，统一为单次原子获取：

```c
// input/input_acq.h — 采集后端统一接口
#pragma once
#include "drv_common.h"
#include "scan_buffer.h"

typedef struct {
    // 启动采集后端（初始化 DMA / 配置 MUX / 启动 SPI 等）
    drv_status_t (*start)(void *ctx);

    // 停止采集后端
    drv_status_t (*stop)(void *ctx);

    // 获取最新一致性快照（单次原子操作，替代 has_fresh_sample + acquire）
    // DRV_OK    = 成功拿到一致性快照，seq 返回当前序号
    // DRV_BUSY  = 当前没有新样本（调用者沿用上次快照或等待通知）
    // DRV_ERROR = 底层采集异常
    drv_status_t (*acquire_latest)(void *ctx, scan_buffer_t *out, uint32_t *seq);

    void *ctx;  // 指向具体后端的上下文
} input_acq_t;
```

`seq` 的三个用途：
1. **判断是否有更新**：`curr_seq == last_seq` → 无新数据，沿用上次
2. **检测丢帧**：`curr_seq - last_seq > 1` → 采集速率快于消费速率
3. **安全链路时效性监控**：L2 可检查 L1 输入的 `seq` 是否停滞

### 4.4 scan_buffer_t — 通用样本载体

```c
// input/scan_buffer.h — 纯数据结构，零外部依赖
#pragma once
#include <stdint.h>

#define SCAN_BUF_MAX_CHANNELS  16

typedef struct {
    uint16_t values[SCAN_BUF_MAX_CHANNELS];  // 各通道原始 ADC 值
    uint8_t  count;                           // 有效通道数
    uint32_t seq;                             // 单调递增样本序号
    uint32_t timestamp_ms;                    // 快照生成时间
} scan_buffer_t;
```

`scan_buffer_t` 是整个输入采集层的**解耦核心**：
- 上游（`input_acq_t`）只写入它
- 下游（输入源）只读取它
- 它本身不依赖任何硬件接口，可在 PC Mock 上直接填充测试数据
- v5 明确：它是**快照对象**，不是 DMA 工作缓冲
- `input_acq_t.acquire_latest()` 必须同时填充 `seq` 与 `timestamp_ms`
- `main_task` 在消费前校验 `now_ms - timestamp_ms <= INPUT_MAX_AGE_MS`

### 4.5 Sensor Backend — 同步拉取实现

**MUX + ADC 后端**（JS-300/JS-400 等使用 MUX 的产品）：

```c
// input/backend/mux_adc_acq.c — MUX 选通 + ADC 逐通道采集
#include "input_acq.h"
#include "adc_reader.h"
#include "drv_gpio.h"

typedef struct {
    const adc_reader_t  *adc;
    const gpio_writer_t *gpio;
    adc_channel_id_t     adc_channel;
    gpio_pin_id_t        sel_pins[4];
    uint8_t              sel_pin_count;
    uint8_t              total_channels;
    uint16_t             settle_us;
    uint32_t             seq;           // 单调递增序号
} mux_adc_acq_ctx_t;

static drv_status_t mux_adc_acquire(void *ctx, scan_buffer_t *out, uint32_t *seq) {
    mux_adc_acq_ctx_t *c = (mux_adc_acq_ctx_t *)ctx;
    out->count = c->total_channels;

    for (uint8_t ch = 0; ch < c->total_channels; ch++) {
        for (uint8_t s = 0; s < c->sel_pin_count; s++) {
            const gpio_level_t level = (ch >> s) & 1u ? GPIO_HIGH : GPIO_LOW;
            c->gpio->write(c->sel_pins[s], level);
        }
        drv_delay_us(c->settle_us);
        uint16_t raw = 0;
        c->adc->read(c->adc_channel, &raw);
        out->values[ch] = raw;
    }

    out->seq = ++c->seq;
    out->timestamp_ms = drv_get_time_ms();
    *seq = out->seq;
    return DRV_OK;  // 同步拉取永远成功（除非硬件错误）
}
```

**直接 ADC 多通道后端**

```c
// input/backend/direct_adc_acq.c
#include "input_acq.h"
#include "adc_reader.h"

typedef struct {
    const adc_reader_t *adc;
    adc_channel_id_t    channels[SCAN_BUF_MAX_CHANNELS];
    uint8_t             channel_count;
    uint32_t            seq;
} direct_adc_acq_ctx_t;

static drv_status_t direct_adc_acquire(void *ctx, scan_buffer_t *out, uint32_t *seq) {
    direct_adc_acq_ctx_t *c = (direct_adc_acq_ctx_t *)ctx;
    out->count = c->channel_count;

    for (uint8_t i = 0; i < c->channel_count; i++) {
        uint16_t raw = 0;
        c->adc->read(c->channels[i], &raw);
        out->values[i] = raw;
    }

    out->seq = ++c->seq;
    out->timestamp_ms = drv_get_time_ms();
    *seq = out->seq;
    return DRV_OK;
}
```

**SPI Hall 传感器后端**（AS5048A 等磁编码器，同步模式）

```c
// input/backend/spi_hall_acq.c — SPI 霍尔角度传感器
#include "input_acq.h"
#include "drv_spi.h"

typedef struct {
    const spi_driver_t *spi;
    uint8_t             cs_pin;
    uint8_t             channel_index;  // 在 scan_buffer 中的位置
    uint32_t            seq;
} spi_hall_acq_ctx_t;

static drv_status_t spi_hall_acquire(void *ctx, scan_buffer_t *out, uint32_t *seq) {
    spi_hall_acq_ctx_t *c = (spi_hall_acq_ctx_t *)ctx;

    uint8_t tx[2] = {0xFF, 0xFF};  // AS5048A: NOP 命令读角度
    uint8_t rx[2] = {0};
    drv_status_t st = c->spi->transfer(c->cs_pin, tx, rx, 2);
    if (st != DRV_OK) return DRV_ERROR;

    uint16_t angle = ((rx[0] & 0x3F) << 8) | rx[1];  // 14-bit 角度
    out->values[c->channel_index] = angle;
    out->count = c->channel_index + 1;

    out->seq = ++c->seq;
    out->timestamp_ms = drv_get_time_ms();
    *seq = out->seq;
    return DRV_OK;
}
```

### 4.6 Sensor Backend — 异步推送实现（DMA / SPI DMA）

DMA 不是扫描策略，而是**采集机制**。v5 使用双缓冲 + 序号 + 内存屏障，上层永远不接触 ISR callback。

```c
// input/backend/dma_adc_acq.c — DMA 连续采集后端
#include "input_acq.h"
#include "adc_continuous.h"
#include "frame_publisher.h"  // 双缓冲帧发布（见 3.7.6）

typedef struct {
    const adc_continuous_t *adc_cont;
    frame_publisher_t       fp;                              // 双缓冲 + 序号
    uint16_t                buf_a[SCAN_BUF_MAX_CHANNELS];    // ping 缓冲
    uint16_t                buf_b[SCAN_BUF_MAX_CHANNELS];    // pong 缓冲
    uint8_t                 channel_count;
    uint32_t                last_consumed_seq;               // 消费者上次读到的序号
} dma_adc_acq_ctx_t;

// 平台层 DMA 完成 ISR 中调用（callback 不暴露给上层）
static void dma_complete_isr(void *ctx) {
    dma_adc_acq_ctx_t *c = (dma_adc_acq_ctx_t *)ctx;
    uint8_t completed_idx = /* 平台 HAL 告知完成的缓冲区索引 */ 0;
    frame_publish(&c->fp, completed_idx);  // 含 async_write_barrier()
}

static drv_status_t dma_adc_start(void *ctx) {
    dma_adc_acq_ctx_t *c = (dma_adc_acq_ctx_t *)ctx;
    // 初始化 frame_publisher
    c->fp.buffers[0] = c->buf_a;
    c->fp.buffers[1] = c->buf_b;
    c->fp.frame_size = c->channel_count * sizeof(uint16_t);
    async_notify_init(&c->fp.notify);
    // 启动 DMA 连续采集
    adc_channel_id_t chs[SCAN_BUF_MAX_CHANNELS];
    // ... 填充通道列表 ...
    return c->adc_cont->start_continuous(chs, c->channel_count,
                                          c->buf_a, c->channel_count);
}

static drv_status_t dma_adc_acquire(void *ctx, scan_buffer_t *out, uint32_t *seq) {
    dma_adc_acq_ctx_t *c = (dma_adc_acq_ctx_t *)ctx;
    uint32_t new_seq;

    // frame_consume 内含 async_read_barrier()
    drv_status_t st = frame_consume(&c->fp, out->values, &new_seq);
    if (st != DRV_OK) return st;

    if (new_seq == c->last_consumed_seq) {
        return DRV_BUSY;  // 无新帧
    }

    out->count = c->channel_count;
    out->seq = new_seq;
    out->timestamp_ms = drv_get_time_ms();
    *seq = new_seq;
    c->last_consumed_seq = new_seq;
    return DRV_OK;
}
```

> **关键设计**：DMA ISR 只做三件事：① 记录完成的缓冲区索引 ② 更新序号 ③ 发通知。不做任何数据拷贝或业务逻辑。上层通过 `acquire_latest()` 从非活跃缓冲区取稳定快照。

### 4.7 buf_axis_source — 缓冲区轴输入源

```c
// input/buf_axis_source.h
#pragma once
#include "drv_common.h"
#include "scan_buffer.h"

    if (c->channel_index >= c->buf->count) {
        return DRV_INVALID_ARG;
    }
    *raw = (int32_t)c->buf->values[c->channel_index];

}
```

### 4.8 buf_btn_source — 缓冲区按键输入源

```c
// input/buf_btn_source.h
#pragma once
#include "drv_common.h"
#include "scan_buffer.h"

// 按键输入源统一接口
typedef struct {
    drv_status_t (*read_buttons)(void *ctx, uint16_t *button_mask);
    void *ctx;
    uint8_t button_count;
} button_source_t;

// 基于 scan_buffer 的按键输入源上下文
// 每个按键占独立 MUX 通道，5V→2.5V 分压，按下时读数 ≥ threshold
typedef struct {
    const scan_buffer_t *buf;
    uint8_t    channel_indices[12];  // 各按键对应的 scan_buffer 通道索引
    uint16_t   threshold;            // 判定为按下的 ADC 阈值（对应 ~2.5V）
    uint8_t    count;                // 按键数量
} buf_btn_ctx_t;

static inline drv_status_t buf_btn_read(void *ctx, uint16_t *mask) {
    const buf_btn_ctx_t *c = (const buf_btn_ctx_t *)ctx;
    *mask = 0;
    for (uint8_t i = 0; i < c->count; i++) {
        if (c->buf->values[c->channel_indices[i]] >= c->threshold) {
            *mask |= (1u << i);
        }
    }
    return DRV_OK;
}
```

阈值检测方式天然支持**多键同按**：每个按键在独立 MUX 通道上有独立的分压电路，互不干扰。

## 五、信号处理管线架构（v4 改进：统一为显式 pipeline，删除泛型 chain）

### 5.1 内部信号表示：无符号原始范围，不归一化

信号处理管线**不做归一化**，全程保持传感器原始范围。值域转换由独立的 `output_adapter` 层在协议打包前一次性完成（见第六章）。

| 传感器 | 原始范围 | 中心值 | 数据类型 |
|--------|---------|--------|---------|
| 12-bit ADC | 0–4095 | ~2048（校准值） | uint16_t |
| 14-bit Hall | 0–16383 | ~8192（校准值） | uint16_t |

**为什么不在管线内归一化：**

- **精度损失是真实的** — 14-bit 传感器抖动 ±2~3 LSB，一次整数除法映射可能把信号淹没在量化噪声中
- **校准不对称时引入非线性** — 正/负向量程不同，归一化会在零点附近产生增益跳变
- **管线各阶段不需要统一量纲** — 故障检测在 `uint16_t` 域，滤波/死区在 `int32_t` 域围绕零点操作，天然不关心绝对范围

**v4 类型域分界：**

```
              uint16_t 域                    int32_t 域
         ┌──────────────────┐    ┌──────────────────────────┐
  raw ──→│ fault_detect     │───→│ calibrate (分界点)        │
         │ 阈值比较          │    │ linear: raw - center     │
         └──────────────────┘    │ circular: wrap_subtract  │
                                 ├──────────────────────────┤
                                 │ lowpass_filter           │
                                 ├──────────────────────────┤
                                 │ deadzone                 │
                                 ├──────────────────────────┤
                                 │ output (有符号偏移量)     │
                                 └──────────────────────────┘
                                          │
                                 output_adapter 按需缩放（唯一映射点）
```

### 5.2 设计动机

原设计中 `joystick_core.c` 的 `process_axis()` 函数同时负责 ADC 采集、故障检测、校准计算、死区处理、按键消抖五个职责。任何一个需求变化（换消抖算法、换死区曲线、加数字滤波）都要修改这个文件。

改进：拆分为信号处理管线（pipeline），每个阶段是独立模块，各自只有一个变更理由。

> **v4 变更：** v3 同时存在泛型 `signal_chain_t`（`int16_t` 统一签名）和显式 `axis_pipeline_t`（`uint16_t→int32_t` 分界）两套接口。v4 删除泛型 chain，只保留显式 pipeline。理由：管线阶段顺序固定，真正变化的是参数和策略，泛型 chain 的灵活性不被需要，反而引入类型不匹配和概念混淆。

### 5.3 各阶段独立实现（v4 统一签名）

```c
// signal/calibrator.c
int32_t calibrate_linear(uint16_t raw, const calibrator_ctx_t *ctx) {
    // 不归一化，直接返回有符号偏移量
    return (int32_t)raw - (int32_t)ctx->center;
}

int32_t calibrate_circular(uint16_t raw, const calibrator_ctx_t *ctx) {
    // 环形传感器（霍尔角度）：处理回绕
    int32_t diff = (int32_t)raw - (int32_t)ctx->center;
    int32_t half_range = (int32_t)(ctx->max - ctx->min) / 2;
    if (diff > half_range)  diff -= (int32_t)(ctx->max - ctx->min);
    if (diff < -half_range) diff += (int32_t)(ctx->max - ctx->min);
    return diff;
}
```

```c
// signal/filter.c
int32_t lowpass_filter(lowpass_ctx_t *ctx, int32_t input) {
    int32_t num = (int32_t)ctx->alpha * ctx->prev_output
                + (256 - (int32_t)ctx->alpha) * input;
    num += (num >= 0) ? 128 : -128;  // 四舍五入，避免长期负偏差
    int32_t out = num / 256;
    ctx->prev_output = out;
    return out;
}
```

### 5.4 显式信号管线（v4 唯一的信号链接口）

管线阶段顺序在产品中基本固定，真正变化的是每个阶段的**参数和策略**。用显式函数实现，类型分界清晰：

```c
// signal/axis_pipeline.h
#pragma once
#include <stdint.h>
#include "fault_detector.h"
#include "calibrator.h"
#include "filter.h"
#include "deadzone.h"

typedef struct {
    uint16_t       fault_threshold_low;
    uint16_t       fault_threshold_high;
    uint8_t        filter_alpha;       // 0 = 不启用滤波
    int32_t        deadzone_threshold;
    deadzone_fn_t  deadzone_fn;        // 死区策略（linear / scurve / exponential）
} axis_pipeline_config_t;

typedef struct {
    // 故障检测上下文
    fault_detect_ctx_t fault_ctx;
    // 校准上下文
    calibrator_ctx_t   cal_ctx;
    calibrate_fn_t     calibrate;     // 线性 / 环形（产品配置注入）
    // 滤波上下文
    lowpass_ctx_t      filter_ctx;
    // 死区
    int32_t            dz_threshold;
    deadzone_fn_t      dz_fn;
    // 输出
    int32_t            output;        // 有符号偏移量（不归一化）
    uint16_t           max_deflection; // 该轴的最大偏转量（range/2），供 output_adapter 使用
} axis_pipeline_t;

void axis_pipeline_init(axis_pipeline_t *p,
                        const axis_pipeline_config_t *cfg,
                        calibrate_fn_t calibrate,
                        uint8_t resolution_bits);
void axis_pipeline_process(axis_pipeline_t *p, uint16_t raw);
```

```c
// signal/axis_pipeline.c
void axis_pipeline_init(axis_pipeline_t *p,
                        const axis_pipeline_config_t *cfg,
                        calibrate_fn_t calibrate,
                        uint8_t resolution_bits) {
    p->fault_ctx.threshold_low  = cfg->fault_threshold_low;
    p->fault_ctx.threshold_high = cfg->fault_threshold_high;
    p->calibrate    = calibrate;
    p->filter_ctx.alpha = cfg->filter_alpha;
    p->filter_ctx.prev_output = 0;
    p->dz_threshold = cfg->deadzone_threshold;
    p->dz_fn        = cfg->deadzone_fn;
    p->output       = 0;
    p->max_deflection = (uint16_t)(1u << (resolution_bits - 1));  // range/2
}

void axis_pipeline_process(axis_pipeline_t *p, uint16_t raw) {
    // 1. 故障检测（uint16_t 域）
    if (fault_detect(&p->fault_ctx, raw)) {
        p->output = 0;
        return;
    }

    // 2. 校准（uint16_t → int32_t，类型分界点）
    //    线性传感器: (int32_t)raw - center
    //    数字角度:   circular_subtract(raw, center)
    int32_t val = p->calibrate(raw, &p->cal_ctx);

    // 3. 低通滤波（int32_t 域）
    if (p->filter_ctx.alpha > 0) {
        val = lowpass_filter(&p->filter_ctx, val);
    }

    // 4. 死区（int32_t 域，围绕零点）
    val = p->dz_fn(val, p->dz_threshold);

    p->output = val;
}
```


## 六、协议层架构（v4 改进：Output Adapter + Profile 驱动 + Protocol Bundle）

### 6.1 设计动机

v3 协议层存在的问题：

1. **`void *config`** — `init(const void *config)` 无类型检查，传错 config 编译通过、运行崩溃 → v3 已用 Protocol Bundle 解决
2. **协议行为 hardcode** — OD 索引、PGN、字节打包、值范围全写死在代码里 → v3 已用 Profile 驱动解决
3. **值域转换嵌入协议引擎** — `canopen_engine_send()` 同时做值缩放和协议打包，违反 SRP → **v4 新增 Output Adapter 层解决**
4. **`joystick_data_t` 非自描述** — `max_deflection` 为单一值，假设所有轴分辨率相同，但 JS-300 混合 12-bit/14-bit → **v4 改为 per-axis**

### 6.3 轴输出映射描述（通用）

所有协议共享的轴值转换描述，定义目标范围和方向：

```c
// output/axis_output_map.h — v4: 从 protocols/ 移至 output/，供 output_adapter 使用
#pragma once
#include <stdint.h>

typedef struct {
    int32_t output_min;     // 目标最小值（如 -10000 / 0）
    int32_t output_max;     // 目标最大值（如 +10000 / 65535）
    int8_t  direction;      // +1 正常, -1 反向
} axis_output_map_t;

// 通用转换：管线输出 → 协议目标范围（纯函数）
static inline int32_t axis_map_output(int32_t val, int32_t max_deflection,
                                       const axis_output_map_t *map) {
    if ((map == NULL) || (max_deflection <= 0)) {
        return 0;
    }

    const int32_t center = (map->output_min + map->output_max) / 2;
    const int32_t half   = (map->output_max - map->output_min) / 2;
    int64_t scaled = (int64_t)val * (int64_t)map->direction * (int64_t)half;
    int64_t out = (int64_t)center + scaled / (int64_t)max_deflection;

    if (out > map->output_max) out = map->output_max;
    if (out < map->output_min) out = map->output_min;
    return (int32_t)out;
}
```

### 6.4 Output Adapter 层（v4 新增）

**设计动机：** v3 中值域转换嵌入协议引擎内部，导致：
- 协议引擎违反 SRP（转换 + 打包两个职责）
- HMI 显示/数据记录/远程调试想用缩放后的值无法复用
- 测试值转换需要 mock CAN 驱动
- `joystick_data_t` 非自描述（`axes[]` 的含义依赖外部上下文）

v4 引入 `output_adapter` 层，职责单一：将管线内部偏移量转换为自描述的输出值。

```c

#include "joystick_data.h"
#include "axis_output_map.h"

// 每轴输出描述：自描述，不依赖外部上下文
typedef struct {
    int32_t  value;          // 转换后的值（目标范围内）
    int32_t  output_min;     // 该值的有效下界
    int32_t  output_max;     // 该值的有效上界
    uint8_t  valid;          // 数据是否有效
} axis_output_t;

typedef struct {
    axis_output_t axes[MAX_AXES];
    uint8_t       axes_count;
    uint16_t      buttons;
    uint8_t       buttons_count;
    uint32_t      timestamp_ms;
    uint8_t       safety_state;
    uint8_t       error_flags;
} output_data_t;

// 纯函数：joystick_data_t → output_data_t
void output_adapt(const joystick_data_t *raw,
                  const axis_output_map_t *maps,
                  uint8_t map_count,
                  output_data_t *out);
```

```c
// output/output_adapter.c
#include "output_adapter.h"

void output_adapt(const joystick_data_t *raw,
                  const axis_output_map_t *maps,
                  uint8_t map_count,
                  output_data_t *out) {
    out->axes_count    = raw->axes_count;
    out->buttons       = raw->buttons;
    out->buttons_count = raw->buttons_count;
    out->timestamp_ms  = raw->timestamp_ms;
    out->safety_state  = raw->safety_state;
    out->error_flags   = raw->error_flags;

    for (uint8_t i = 0; i < map_count && i < raw->axes_count; i++) {
        const axis_output_map_t *m = &maps[i];
        out->axes[i].output_min = m->output_min;
        out->axes[i].output_max = m->output_max;
        out->axes[i].valid      = (raw->axes_valid >> i) & 1u;
        out->axes[i].value = out->axes[i].valid
                           ? axis_map_output(raw->axes[i],
                                             (int32_t)raw->max_deflection[i], m)
                           : ((m->output_min + m->output_max) / 2);
    }
}
```

**层级职责对照（v4）：**

```
层                    输入类型         输出类型         职责
─────────────────────────────────────────────────────────────
axis_pipeline         uint16_t raw     int32_t offset   信号处理（故障/校准/滤波/死区）
joystick_data_t       —                —                管线内部数据合同（原始偏移量）
output_adapter        joystick_data_t  output_data_t    值域转换（纯函数，可复用）
protocol_engine       output_data_t    CAN frame        协议打包（OD/PGN/raw frame）
```

**设计模式对照：**

| 模式 | v3 | v4 |
|------|------|--------|
| **SRP** | 协议层兼做转换+打包 | 转换和打包分离 |
| **Adapter** | 转换逻辑嵌入协议引擎内部 | `output_adapter` 是独立 Adapter |
| **Anti-Corruption Layer** | 协议层直接解释管线内部表示 | `output_data_t` 是自描述的边界对象 |
| **可复用性** | HMI/日志/调试想用缩放后的值需重复代码 | `output_adapt()` 是纯函数，任何消费者可调用 |
| **可测试性** | 测试值转换需要 mock CAN 驱动 | `output_adapt()` 纯函数，直接单元测试 |

### 6.6 CANopen 通用引擎（v4 改进：只管打包，不做值转换）

```c
// protocols/canopen/canopen_engine.c
#include "canopen_profile.h"
#include "canopen_stack.h"
#include "output_adapter.h"     // v4: 接收 output_data_t

static const canopen_profile_t *profile;
static const can_port_t *can;
static uint8_t node_id;

drv_status_t canopen_engine_init(const can_port_t *port,
                                 const canopen_profile_t *cfg_profile,
                                 uint8_t cfg_node_id) {
    can = port;
    node_id = cfg_node_id;
    profile = cfg_profile;

    co_node_init(node_id);
    co_nmt_set_heartbeat(profile->heartbeat_ms);

    // 根据 profile 初始化 OD 默认值
    od_load_defaults(profile->od);

    // 根据 profile 配置 TPDO 映射
    for (uint8_t t = 0; t < profile->tpdo_count; t++) {
        const tpdo_config_t *tp = &profile->tpdos[t];
        for (uint8_t e = 0; e < tp->count; e++) {
            co_tpdo_map(t + 1, tp->entries[e].od_index,
                        tp->entries[e].od_subindex,
                        tp->entries[e].bit_length);
        }
        co_tpdo_set_event_timer(t + 1, tp->event_timer_ms);
    }

    event_bus_subscribe(EVT_SAFETY_STATE_CHANGED, on_safety_changed);
    return DRV_OK;
}

// v4: 接收 output_data_t（已经过 output_adapter 转换，值已在目标范围内）
drv_status_t canopen_engine_send(const output_data_t *data) {
    // 直接将已转换的值写入 OD，不再调用 axis_map_output()
    for (uint8_t i = 0; i < profile->axis_map_count && i < data->axes_count; i++) {
        od_update_value(profile->od,
                        profile->tpdos[0].entries[i].od_index,
                        profile->tpdos[0].entries[i].od_subindex,
                        data->axes[i].value);
    }
    // 按键写入 OD
    od_update_value(profile->od, profile->digital_input_index,
                    profile->digital_input_subindex, data->buttons);
    return DRV_OK;
}

drv_status_t canopen_engine_on_rx_frame(const can_frame_t *frame) {
    return co_node_on_frame(frame);
}

drv_status_t canopen_engine_process_periodic(uint32_t now_ms) {
    (void)now_ms;
    co_node_process();
    return DRV_OK;
}

static void on_safety_changed(const event_t *evt) {
    if (evt->data.safety_state == SAFETY_STOP) {
        co_emcy_send(profile->emcy_code_sensor_fault);
    }
}
```

### 6.7 J1939 Profile — PGN + SPN 描述符

J1939 没有 OD，变化点是 PGN 选择和帧内字节打包方式。用 SPN 描述符定义：

```c
// protocols/j1939/j1939_profile.h
#pragma once
#include <stdint.h>

typedef enum {
    SPN_SOURCE_AXIS,       // 来源：轴数据
    SPN_SOURCE_BUTTON,     // 来源：按键位掩码
    SPN_SOURCE_SAFETY,     // 来源：安全状态
    SPN_SOURCE_CONST,      // 来源：固定值（如 0xFF 填充）
} spn_source_t;

typedef struct {
    spn_source_t source;
    uint8_t  source_index;    // 轴/按键编号（CONST 时为固定值）
    uint8_t  byte_offset;     // 帧中起始字节（0-7）
    uint8_t  bit_offset;      // 字节内起始位（0-7）
    uint8_t  bit_length;      // 位宽
    int32_t  scale_num;       // 缩放分子
    int32_t  scale_den;       // 缩放分母（physical = raw * num / den + offset）
    int32_t  add_offset;      // 偏移量
    int8_t   direction;       // +1 正常, -1 反向
} spn_descriptor_t;

typedef struct {
    uint32_t pgn;
    uint8_t  priority;
    uint16_t tx_interval_ms;
    const spn_descriptor_t *spns;
    uint8_t  spn_count;
} pgn_descriptor_t;

typedef struct {
    const char             *profile_name;
    uint8_t                 source_address;
    const pgn_descriptor_t *pgns;
    uint8_t                 pgn_count;
} j1939_profile_t;
```

### 6.8 J1939 通用引擎（v4 改进：接收 output_data_t）

```c
// protocols/j1939/j1939_engine.c
#include "j1939_profile.h"
#include "output_adapter.h"     // v4: 接收 output_data_t

static const j1939_profile_t *profile;
static const can_port_t *can;

drv_status_t j1939_engine_init(const can_port_t *port,
                               const j1939_profile_t *cfg_profile) {
    can = port;
    profile = cfg_profile;
    return DRV_OK;
}

// v4: 接收 output_data_t，轴值已经过 output_adapter 转换
drv_status_t j1939_engine_send(const output_data_t *data) {
    for (uint8_t p = 0; p < profile->pgn_count; p++) {
        const pgn_descriptor_t *pgn = &profile->pgns[p];
        can_frame_t frame = {0};
        frame.id = ((uint32_t)pgn->priority << 26)
                 | (pgn->pgn << 8)
                 | profile->source_address;
        frame.is_extended = 1;
        frame.dlc = 8;
        memset(frame.data, 0xFF, 8);

        for (uint8_t s = 0; s < pgn->spn_count; s++) {
            const spn_descriptor_t *spn = &pgn->spns[s];
            int32_t val = spn_get_source(data, spn);
            // J1939 SPN 仍需自己的 scale/offset 打包（协议特有的位级编码）
            uint32_t packed = (uint32_t)(val * spn->direction
                              * spn->scale_num / spn->scale_den
                              + spn->add_offset);
            spn_write_bits(frame.data, packed,
                           spn->byte_offset, spn->bit_offset, spn->bit_length);
        }
        can->send(can->ctx, &frame);
    }
    return DRV_OK;
}

static int32_t spn_get_source(const output_data_t *data,
                              const spn_descriptor_t *spn) {
    switch (spn->source) {
        case SPN_SOURCE_AXIS:
            if ((spn->source_index >= data->axes_count) ||
                !data->axes[spn->source_index].valid) {
                return 0;
            }
            return data->axes[spn->source_index].value;
        case SPN_SOURCE_BUTTON:  return data->buttons;
        case SPN_SOURCE_SAFETY:  return data->safety_state;
        case SPN_SOURCE_CONST:   return spn->source_index;
        default:                 return 0;
    }
}

drv_status_t j1939_engine_on_rx_frame(const can_frame_t *frame) {
    (void)frame;
    return DRV_NOT_SUPPORTED;
}

drv_status_t j1939_engine_process_periodic(uint32_t now_ms) {
    (void)now_ms;
    return DRV_OK;
}
```

### 6.10 协议能力与运行时接口（v5 改进：send / on_rx_frame / process_periodic 三分接口）

```c
// protocols/protocol_runtime.h
#include "output_adapter.h"     // v4: send 接收 output_data_t

typedef enum {
    PROTO_CAP_RECEIVABLE    = (1 << 0),
    PROTO_CAP_CONFIGURABLE  = (1 << 1),
    PROTO_CAP_DIAGNOSABLE   = (1 << 2),
```

初始化时根据 bundle type 构建运行时接口（唯一的 switch，启动时执行一次）：

```c
// protocols/protocol_init.c
static protocol_runtime_t runtime;

drv_status_t protocol_init(const protocol_bundle_t *bundle) {
    runtime.type = bundle->type;

    switch (bundle->type) {
        case PROTOCOL_TYPE_CANOPEN:
            canopen_engine_init(bundle->canopen.can,
                                bundle->canopen.profile,
                                bundle->canopen.node_id);
            runtime.send         = canopen_engine_send;
            runtime.on_rx_frame  = canopen_engine_on_rx_frame;
            runtime.process_periodic = canopen_engine_process_periodic;
            runtime.get_bus_diag = canopen_engine_get_diag;
            runtime.capabilities = PROTO_CAP_RECEIVABLE
                                 | PROTO_CAP_CONFIGURABLE
                                 | PROTO_CAP_DIAGNOSABLE;
            break;

        case PROTOCOL_TYPE_J1939:
            j1939_engine_init(bundle->j1939.can, bundle->j1939.profile);
            runtime.send         = j1939_engine_send;
            runtime.on_rx_frame  = j1939_engine_on_rx_frame;
            runtime.process_periodic = j1939_engine_process_periodic;
            runtime.get_bus_diag = NULL;
            runtime.capabilities = PROTO_CAP_RECEIVABLE;
            break;

        case PROTOCOL_TYPE_CAN_RAW:
            can_raw_engine_init(&bundle->can_raw);
            runtime.send         = can_raw_engine_send;
            runtime.on_rx_frame  = can_raw_engine_on_rx_frame;
            runtime.process_periodic = can_raw_engine_process_periodic;
            runtime.get_bus_diag = NULL;
            runtime.capabilities = PROTO_CAP_RECEIVABLE;
            break;

        default:
            return DRV_INVALID_ARG;
    }
    return DRV_OK;
}

const protocol_runtime_t* protocol_get_runtime(void) {
    return &runtime;
}
```

## 八、事件总线（v5 重写：异步分发 + 双优先级队列 + 背压策略）

### 8.1 设计动机

原设计中协议插件和安全监控之间没有通信机制。实际场景需求：
- CAN Bus-Off 发生时 → 安全监控需要被通知
- 安全状态变化时 → 协议层需要发送紧急消息
- 传感器故障时 → 诊断模块需要记录日志

v4 的 `event_bus_publish()` 是**同步**调用所有 handler，存在以下风险：
- 慢 handler（如 `co_emcy_send()` 等待 CAN 缓冲区）阻塞发布者
- 安全事件和诊断事件同等对待，队列满时无优先级区分
- `event_bus_init()` 在第八章和第十一章存在两套签名（无参 vs 带 backend），文档漂移

v5 改为"**发布入队、ctrl_task 统一分发**"模型，彻底消除同步阻塞。

### 8.2 事件定义与优先级

```c
// event/event_bus.h — v5: 异步分发 + 双优先级队列 + 背压策略
#pragma once
#include <stdint.h>
#include <stdbool.h>

typedef enum {
    EVT_SENSOR_FAULT,
    EVT_CAN_BUS_OFF,
    EVT_CAN_RECOVERED,
    EVT_SAFETY_STATE_CHANGED,
    EVT_CALIBRATION_COMPLETE,
    EVT_WATCHDOG_WARNING,
    EVT_DEVICE_STATE_CHANGED,
    EVT_MUTEX_TIMEOUT,
    EVT_MAX
} event_type_t;

typedef struct {
    event_type_t type;
    uint32_t     timestamp;
    union {
        uint8_t  sensor_id;
        uint8_t  safety_state;  // 0=normal, 1=degraded, 2=safe-stop
        uint32_t error_code;
        uint8_t  new_device_state;
    } data;
} event_t;

// ---- v5 新增：事件优先级 ----
typedef enum {
    EVT_PRIO_SAFETY = 0,   // 安全/故障事件 → critical_queue
    EVT_PRIO_FAULT  = 1,   // 故障事件 → critical_queue
    EVT_PRIO_STATE  = 2,   // 状态切换 → normal_queue
    EVT_PRIO_DIAG   = 3    // 诊断事件 → normal_queue
} event_priority_t;

// ---- v5 新增：发布返回状态（背压反馈） ----
typedef enum {
    EVT_POST_OK,            // 成功入队
    EVT_POST_DROPPED,       // 队列满，事件被丢弃
    EVT_POST_OVERWROTE_OLD  // 队列满，覆盖了最旧的同级事件
} event_post_status_t;

typedef void (*event_handler_t)(const event_t *evt);

// ---- 队列后端（由平台层注入，支持跨平台） ----
typedef struct {
    bool (*enqueue)(const event_t *evt, event_priority_t prio);
    bool (*dequeue)(event_t *evt);
} event_queue_backend_t;

#define EVENT_BUS_MAX_SUBSCRIBERS 4  // 每个事件类型最多 4 个订阅者

// v5: 统一初始化签名（删除旧版无参 event_bus_init(void)）
void event_bus_init(const event_queue_backend_t *backend);

void event_bus_subscribe(event_type_t type, event_handler_t handler);

// v5: 异步发布 — 写入队列，不直接调用 handler
event_post_status_t event_bus_post(const event_t *evt, event_priority_t prio);

// v5: 由 ctrl_task 周期调用，从队列出队并分发给订阅者
void event_bus_dispatch(void);
```

### 8.3 双优先级队列与背压策略

```
┌──────────────────────────────────────────────────────────────────┐
│                事件总线 v5 架构                                     │
│                                                                    │
│  发布者（任意任务/ISR）                                              │
│       │                                                            │
│       │ event_bus_post(evt, prio)                                  │
│       ▼                                                            │
│  ┌─────────────────┐  ┌─────────────────┐                         │
│  │ critical_queue   │  │ normal_queue     │                         │
│  │ (SAFETY/FAULT)   │  │ (STATE/DIAG)     │                         │
│  │ 深度: 8          │  │ 深度: 16          │                         │
│  └────────┬────────┘  └────────┬────────┘                         │
│           │ 优先出队            │ 次优先出队                         │
│           └──────┬─────────────┘                                   │
│                  ▼                                                  │
│           ctrl_task: event_bus_dispatch()                           │
│                  │                                                  │
│                  ▼                                                  │
│           subscribers[evt->type].handlers[]                         │
└──────────────────────────────────────────────────────────────────┘
```

**满队列策略**：

| 队列 | 满时行为 | 理由 |
|------|---------|------|
| `critical_queue` | 覆盖最旧的非安全事件；若全是安全事件则直接 `safe_shutdown_activate()` | 安全事件绝不静默丢弃 |
| `normal_queue` | 丢弃最新的诊断类事件，累加 `drop_counter` | 诊断事件可容忍丢失 |

`diag_task` 周期上报队列水位和 `drop_counter`，用于监控事件系统健康度。

### 8.4 事件总线实现

```c
// event/event_bus.c — v5: 异步分发实现
#include "event_bus.h"

static const event_queue_backend_t *g_backend;
static uint32_t g_drop_counter;

static struct {
    event_handler_t handlers[EVENT_BUS_MAX_SUBSCRIBERS];
    uint8_t count;
} subscribers[EVT_MAX];

void event_bus_init(const event_queue_backend_t *backend) {
    g_backend = backend;
    g_drop_counter = 0;
    for (uint8_t i = 0; i < EVT_MAX; i++) {
        subscribers[i].count = 0;
    }
}

void event_bus_subscribe(event_type_t type, event_handler_t handler) {
    if (type < EVT_MAX && subscribers[type].count < EVENT_BUS_MAX_SUBSCRIBERS) {
        subscribers[type].handlers[subscribers[type].count++] = handler;
    }
}

event_post_status_t event_bus_post(const event_t *evt, event_priority_t prio) {
    if (g_backend && g_backend->enqueue(evt, prio)) {
        return EVT_POST_OK;
    }
    // 队列满处理
    if (prio <= EVT_PRIO_FAULT && evt->type == EVT_SAFETY_STATE_CHANGED) {
        // 安全事件入队失败 = 最后防线
        safe_shutdown_activate();
    }
    g_drop_counter++;
    return EVT_POST_DROPPED;
}

void event_bus_dispatch(void) {
    event_t evt;
    // 出队并分发（ctrl_task 上下文，非 ISR）
    while (g_backend && g_backend->dequeue(&evt)) {
        if (evt.type < EVT_MAX) {
            for (uint8_t i = 0; i < subscribers[evt.type].count; i++) {
                subscribers[evt.type].handlers[i](&evt);
            }
        }
    }
}
```

### 8.5 Handler 约束与异步化模式

**硬约束**：所有 event handler 必须在 **50μs** 内返回。违反此约束的操作应改为 flag-and-defer 模式。

```c
// 示例：CANopen 协议的 flag-and-defer 模式
// handler 不直接发 CAN，只设标志
static volatile uint8_t pending_emcy = 0;

static void on_safety_changed(const event_t *evt) {
    if (evt->data.safety_state == 2) {  // SAFE_STOP
        pending_emcy = 1;               // 设置标志，不阻塞
    }
}

// proto_task 在自己的上下文中发送
drv_status_t canopen_engine_process_periodic(uint32_t now_ms) {
    (void)now_ms;
    if (pending_emcy) {
        pending_emcy = 0;
        co_emcy_send(profile->emcy_code_sensor_fault);  // 在 proto_task 上下文发送
    }
    co_node_process();
    return DRV_OK;
}
```

### 9.1 设计动机

原设计缺少明确的设备生命周期状态管理。实际嵌入式产品需要管理启动自检、预操作、正常运行、安全停止、校准、固件更新等状态。

### 9.2 状态机定义

```c
// app/device_state.h
#pragma once
#include "event_bus.h"

typedef enum {
    STATE_BOOT,            // 启动自检
    STATE_PRE_OPERATIONAL, // 初始化完成，等待使能
    STATE_OPERATIONAL,     // 正常工作
    STATE_SAFE_STOP,       // 安全停止（故障触发）
    STATE_CALIBRATION,     // 校准模式
    STATE_FIRMWARE_UPDATE, // 固件更新
    STATE_MAX
} device_state_t;

typedef struct {
    device_state_t current;
    drv_status_t (*on_enter)(void);
    drv_status_t (*on_execute)(void);
    drv_status_t (*on_exit)(void);

    // 状态转移表
    struct {
        event_type_t   trigger;
        device_state_t next;
        bool (*guard)(void);  // 转移守卫条件
    } transitions[8];
    uint8_t transition_count;
} state_handler_t;

void device_state_init(void);
void device_state_process(void);
device_state_t device_state_get_current(void);
```

```c
// app/device_state.c

static state_handler_t state_table[STATE_MAX];
static device_state_t  current_state = STATE_BOOT;

void device_state_init(void) {
    // 注册状态处理器
    state_table[STATE_BOOT] = (state_handler_t){
        .on_enter   = boot_self_test,
        .on_execute = boot_check_complete,
        .transitions = {
            { .trigger = EVT_DEVICE_STATE_CHANGED, .next = STATE_PRE_OPERATIONAL,
              .guard = boot_test_passed },
            { .trigger = EVT_SENSOR_FAULT, .next = STATE_SAFE_STOP },
        },
        .transition_count = 2,
    };

    state_table[STATE_OPERATIONAL] = (state_handler_t){
        .on_enter   = operational_enter,
        .on_execute = operational_run,
        .on_exit    = operational_exit,
        .transitions = {
            { .trigger = EVT_SAFETY_STATE_CHANGED, .next = STATE_SAFE_STOP },
            { .trigger = EVT_CALIBRATION_COMPLETE, .next = STATE_CALIBRATION,
              .guard = calibration_requested },
        },
        .transition_count = 2,
    };

    // ... 其他状态配置

    // 订阅所有可能触发转移的事件
    event_bus_subscribe(EVT_SAFETY_STATE_CHANGED, on_state_event);
    event_bus_subscribe(EVT_SENSOR_FAULT, on_state_event);
    event_bus_subscribe(EVT_DEVICE_STATE_CHANGED, on_state_event);
}

static void on_state_event(const event_t *evt) {
    state_handler_t *handler = &state_table[current_state];
    for (uint8_t i = 0; i < handler->transition_count; i++) {
        if (handler->transitions[i].trigger == evt->type) {
            if (!handler->transitions[i].guard
                || handler->transitions[i].guard()) {
                // 执行状态转移
                if (handler->on_exit) handler->on_exit();
                current_state = handler->transitions[i].next;
                state_handler_t *next = &state_table[current_state];
                if (next->on_enter) next->on_enter();
                break;
            }
        }
    }
}

void device_state_process(void) {
    if (state_table[current_state].on_execute) {
        state_table[current_state].on_execute();
    }
}
```

```c
    // 4. 根据通道映射表组装输入源（物理通道 → 逻辑功能）
    for (uint8_t i = 0; i < product_config.hw->axis_map_count; i++) {
        const axis_channel_map_t *m = &product_config.hw->axis_map[i];
        joystick.axis_sources[m->logical_index] =
            create_buf_axis_source(&scan_buf, m->physical_channel);
        joystick.axis_calibrate[m->logical_index] = m->calibrate;
    }
    for (uint8_t i = 0; i < product_config.hw->btn_map_count; i++) {
        const btn_channel_map_t *m = &product_config.hw->btn_map[i];
        joystick.btn_physical_map[m->logical_index] = m->physical_channel;
    }
    joystick.axes_count = product_config.hw->axis_map_count;
    joystick.btns_count = product_config.hw->btn_map_count;

    // 5. 初始化摇杆信号管线
    joystick_init(&joystick, &product_config);

    // 6. 初始化协议（Protocol Bundle，类型安全，无 void*）
    protocol_init(product_config.protocol->bundle);

    // 7. v5: 编译时一致性校验（消除配置双源风险）
    #if defined(SAFETY_ENABLED) && (SAFETY_ENABLED == 1)
        BUILD_ASSERT(product_config.safety != NULL,
            "Safety build requires safety profile in product_config");
    #else
        BUILD_ASSERT(product_config.safety == NULL,
            "Non-safety build must not include safety profile");
    #endif

    // 8. 如果是安全型号，初始化 E-Gas 三层安全架构
    if (product_config.safety != NULL) {
        const safety_channel_config_t *sc = product_config.safety->channel_cfg;

        safety_channel_init(&l2_channel, &(safety_channel_init_cfg_t){
            .adc             = platform_get_adc_voltage(),
            .channel_indices = sc->l2_adc_channels,
            .axes_count      = product_config.hw->axis_map_count,
            .center          = sc->l2_center,
        });

        safety_comparator_init(&l2_comparator, &(safety_comparator_cfg_t){
            .tolerance       = sc->l2_tolerance,
            .fault_threshold = sc->l2_fault_threshold,
        });

        safety_wdg_init(&l2_wdg);
    }

    // 9. 初始化诊断
    diagnostics_init();
}

input_acq_t*           app_get_input_acq(void)      { return &input_acq; }  // v5
```

## 十一、RTOS 任务架构

### 11.1 设计动机

工业摇杆固件必须使用 RTOS，原因：

| 需求 | 裸循环问题 | RTOS 解决方案 |
|------|-----------|--------------|
| 安全监控 2ms 响应 | 被协议处理 or 信号链阻塞 | 高优先级任务抢占 |
| CANopen 心跳 1ms 精度 | 主循环抖动大 | 独立任务 + 硬件定时器 |
| 看门狗喂狗时序 | 长处理链导致超时 | 独立高优先级任务 |
| CAN 接收实时性 | 轮询方式丢帧 | ISR + 队列 + 任务 |

### 11.2 RTOS 抽象层 — 采用 CMSIS-RTOS v2

不自研抽象层，直接使用 ARM 官方 **CMSIS-RTOS v2** 标准 API。原因：

| 对比项 | 自研 os_abstraction.h | CMSIS-RTOS v2 |
|--------|----------------------|---------------|
| API 覆盖 | 需手动实现每个 RTOS 适配 | FreeRTOS / ThreadX / RTX5 均有官方适配 |
| ISR 安全 | 需手动区分 `_from_isr` 版本 | 自动检测 IPSR 寄存器，同一 API 通用 |
| 绝对延时 | 未定义（导致周期漂移） | `osDelayUntil()` 已内置 |
| 优先级继承 | 需额外约定 | `osMutexNew()` 直接传 `osMutexPrioInherit` |
| 平台兼容 | STM32 / S32K 需各写一套 | STM32 CubeMX 默认生成；S32K FreeRTOS 直接支持 |
| 维护成本 | 自维护 | ARM 官方维护，零成本 |
| MISRA 合规 | 需自行保证 | CMSIS 头文件已通过 MISRA 扫描 |

项目中使用的核心 API：


### 11.5 各任务职责（v5 重构：控制面拆分 + 协议栈单线程化）

```c
// safety_task — v5: 最小安全闭环（仅 L2 采集 + 比对 + 喂狗 + 分片诊断）
// 规则：禁止使用 mutex，只通过 atomic/volatile 读取共享数据
// WCET 预算：L2 采样+计算 200μs，比较器 50μs，WDG 20μs，诊断分片 300μs，裕量 >50%
static void safety_task_entry(void *param) {
    safety_channel_ctx_t *l2_ch  = app_get_l2_channel();
    safety_comparator_t  *l2_cmp = app_get_l2_comparator();
    safety_wdg_ctx_t     *l2_wdg = app_get_l2_wdg();
    uint32_t prev_tick = osKernelGetTickCount();

    for (;;) {
        // ── Level 2: 独立传感器采集 + 简化计算 ──
        safety_channel_output_t l2_out;
        safety_channel_compute(l2_ch, &l2_out);

        // ── Level 2: 读取 Level 1 输出并比对 ──
        int32_t l1_axes[MAX_AXES];
        safety_shared_read_l1_output(l1_axes);  // atomic 读取

        compare_result_t cmp = safety_compare(
            l2_cmp, l1_axes, &l2_out, l2_ch->axes_count);

        switch (cmp) {
            case COMPARE_OK:
                safety_wdg_update_token(l2_wdg);   // 允许喂狗
                break;
            case COMPARE_DEGRADED:
                event_bus_post(&(event_t){
                    .type = EVT_SAFETY_STATE_CHANGED,
                    .data.safety_state = 1 }, EVT_PRIO_SAFETY);  // 异步入队
                break;
            case COMPARE_FAIL:
                safe_shutdown_activate();             // L3: 直接硬件关断
                break;
        }

        // ── Level 3: 条件喂狗 ──
        safety_wdg_service(l2_wdg);

        // ── 附加诊断（分片执行，受预算约束）──
        safety_diagnostics_step(g_diag_step++, 300 /*budget_us*/);

        // v5: device_state_process() 和 event_bus_dispatch() 已移至 ctrl_task
        // safety_task 不再执行任何控制面逻辑

        osDelayUntil(prev_tick += 2);  // 绝对延时，保证恒定 2ms 周期
    }
}

// ctrl_task — v5 新增：控制面事务（优先级 AboveNormal，5ms 周期）
// 承接原 safety_task 中的 device_state_process() 和事件分发
static void ctrl_task_entry(void *param) {
    uint32_t prev_tick = osKernelGetTickCount();
    for (;;) {
        device_state_process();
        event_bus_dispatch();   // 从双优先级队列出队并分发
        osDelayUntil(prev_tick += 5);
    }
}

// main_task — Level 1 功能通道（v5: input_acq_t 替代 board_scanner_t）
static void main_task_entry(void *param) {
    joystick_t *js         = app_get_joystick();
    input_acq_t *acq       = app_get_input_acq();     // v5: 替代 board_scanner_t
    scan_buffer_t *buf     = app_get_scan_buf();
    joystick_data_t data;
    output_data_t   out;
    uint32_t acq_seq = 0;
    uint32_t prev_tick = osKernelGetTickCount();

    for (;;) {
        cf_marker_checkpoint(CF_MAIN_START);

        if (device_state_get_current() == STATE_OPERATIONAL) {
            // 1. v5: 通过 input_acq_t 获取一致性快照（原子操作 + 序号）
            uint32_t new_seq;
            drv_status_t st = acq->acquire_latest(acq->ctx, buf, &new_seq);

            if (st == DRV_OK) {
                // 检测丢帧（可选：诊断用途）
                if (new_seq - acq_seq > 1) {
                    // 记录丢帧事件（不阻塞主循环）
                }
                acq_seq = new_seq;
            }
            // DRV_BUSY: 沿用上次快照继续处理（同步后端不会出现此情况）
            uint32_t now_ms = osKernelGetTickCount();
            if ((buf->timestamp_ms != 0u) &&
                ((now_ms - buf->timestamp_ms) > INPUT_MAX_AGE_MS)) {
                event_bus_post(&(event_t){ .type = EVT_INPUT_STALE }, EVT_PRIO_FAULT);
                osDelayUntil(prev_tick += 10);
                continue;
            }
            cf_marker_checkpoint(CF_MAIN_SCAN_DONE);

            // 2. 摇杆更新：buf_axis_source / buf_btn_source 从 buffer 读取
            joystick_update(js);
            joystick_get_data(js, &data);
            cf_marker_checkpoint(CF_MAIN_SIGNAL_DONE);

            // 3. 写入共享比较区（供 Level 2 比对）
            safety_shared_write_l1_output(data.axes, data.axes_count,
                                          buf->seq, buf->timestamp_ms);

            // 4. output_adapter 转换（纯函数）
            output_adapt(&data, product_config.protocol->output_maps,
                         product_config.protocol->output_map_count, &out);

            // 5. v5: main_task 不直接触碰协议内部状态，只投递到 proto_tx_queue
            if (osMessageQueuePut(g_proto_tx_queue, &out, 0, 0) != osOK) {
                event_bus_post(&(event_t){ .type = EVT_PROTO_TX_DROPPED }, EVT_PRIO_DIAG);
            }
        }

        cf_marker_checkpoint(CF_MAIN_END);
        osDelayUntil(prev_tick += 10);  // 绝对延时，保证恒定 10ms 周期
    }
}

// proto_task — 协议栈实时处理（v5: can_port_t.wait_recv 替代回调模式）
static void proto_task_entry(void *param) {
    const protocol_runtime_t *proto = protocol_get_runtime();
    const can_port_t *can = app_get_can_port();     // v5: 替代直接回调
    can_frame_t rx_frame;
    output_data_t tx_out;

    for (;;) {
        while (osMessageQueueGet(g_proto_tx_queue, &tx_out, NULL, 0) == osOK) {
            proto->send(&tx_out);
        }

        // v5: 阻塞等待 CAN 帧（平台层内部处理 ISR→队列转换）
        // 超时 1ms 保证 CANopen 心跳/NMT 定时处理
        if (can->wait_recv(can->ctx, &rx_frame, 1) == DRV_OK) {
            proto->on_rx_frame(&rx_frame);
        }
        proto->process_periodic(osKernelGetTickCount());
    }
}

// diag_task — 低优先级后台服务（v4 无变化）
static void diag_task_entry(void *param) {
    uint32_t prev_tick = osKernelGetTickCount();

    for (;;) {
        diagnostics_update();
        nv_storage_flush();   // 将缓存写入 Flash/EEPROM
        logging_flush();
        task_health_check();  // 检查各任务 alive 标记
        osDelayUntil(prev_tick += 100);  // 绝对延时，保证恒定 100ms 周期
    }
}
```

### 11.7 事件总线架构（v5 改进：控制面/数据面分离 + 队列后端注入）

#### 11.7.1 控制面与数据面分离原则

v4 的 event_bus 同时承载状态事件和高频数据，v5 明确分离：

| 面 | 内容 | 传输机制 | 频率 |
|----|------|---------|------|
| **控制面** | 故障事件、状态切换、模式变更、CAN Bus-Off | `event_bus`（观察者模式） | 低频（事件驱动） |
| **数据面** | ADC 样本、CAN RX 帧、SPI 传感器值 | `input_acq_t` / `can_port_t`（快照+队列） | 高频（周期性） |

**硬约束**：event_bus **不承载高频原始数据**。ADC/CAN/SPI 的周期性样本走 `input_acq_t.acquire_latest()` 或 `can_port_t.wait_recv()`。

#### 11.7.2 ISR 安全：队列后端注入

v4 的 `event_bus_publish_from_isr()` 依赖 CMSIS-RTOS v2 的 IPSR 寄存器自动检测（ARM Cortex-M 专属）。v5 将队列后端改为可注入策略，支持跨平台：

```c
// event/event_bus.h — v5: 队列后端注入

// 事件队列后端（由平台层注入，支持跨平台；与第八章 event_bus_init 签名完全一致）
// 注意：v5 删除了旧版 event_bus_init(void) 和 event_bus_publish() 同步接口
// 统一使用 event_bus_post() 异步入队 + event_bus_dispatch() 在 ctrl_task 中分发
typedef struct {
    bool (*enqueue)(const event_t *evt, event_priority_t prio);  // ISR/任务通用入队
    bool (*dequeue)(event_t *evt);                                // 任务上下文出队
} event_queue_backend_t;

// v5 统一签名（见第八章 8.2）
void event_bus_init(const event_queue_backend_t *backend);

// 异步发布：写入双优先级队列（ISR 和任务上下文均可调用）
event_post_status_t event_bus_post(const event_t *evt, event_priority_t prio);

// 由 ctrl_task 周期调用，从队列出队并分发给订阅者
void event_bus_dispatch(void);
```

各平台实现：

```c
// platform/stm32g4/event_queue_cmsis.c — ARM 平台（v5: 双队列）
#include "cmsis_os2.h"
static osMessageQueueId_t critical_queue;  // 深度 8
static osMessageQueueId_t normal_queue;    // 深度 16

static bool cmsis_enqueue(const event_t *evt, event_priority_t prio) {
    osMessageQueueId_t q = (prio <= EVT_PRIO_FAULT) ? critical_queue : normal_queue;
    return osMessageQueuePut(q, evt, 0, 0) == osOK;
}
static bool cmsis_dequeue(event_t *evt) {
    // 优先出队 critical_queue
    if (osMessageQueueGet(critical_queue, evt, NULL, 0) == osOK) return true;
    return osMessageQueueGet(normal_queue, evt, NULL, 0) == osOK;
}
const event_queue_backend_t event_queue_cmsis = {
    .enqueue = cmsis_enqueue,
    .dequeue = cmsis_dequeue,
};

// platform/test/event_queue_mock.c — PC 测试平台（v5: 简化双队列模拟）
static event_t crit_buf[8], norm_buf[16];
static uint8_t crit_head, crit_tail, norm_head, norm_tail;
static bool mock_enqueue(const event_t *evt, event_priority_t prio) {
    if (prio <= EVT_PRIO_FAULT) {
        crit_buf[crit_head++ & 7] = *evt;
    } else {
        norm_buf[norm_head++ & 15] = *evt;
    }
    return true;
}
static bool mock_dequeue(event_t *evt) {
    if (crit_head != crit_tail) { *evt = crit_buf[crit_tail++ & 7]; return true; }
    if (norm_head != norm_tail) { *evt = norm_buf[norm_tail++ & 15]; return true; }
    return false;
}
const event_queue_backend_t event_queue_mock = {
    .enqueue = mock_enqueue,
    .dequeue = mock_dequeue,
};
```

> **安全约束不变**：L2 安全通道不依赖 event_bus，最多读共享快照或独立采样。L3 硬件关断路径（GPIO 直驱安全继电器）完全不经 event_bus/RTOS。

### 11.8 时序要求与优先级反转防范

```
时间线（最坏情况）：

0ms ─── safety_task 开始执行 (2ms 周期)
         安全监控 + 看门狗喂狗
0.3ms ── safety_task 完成，让出 CPU
         proto_task 获得 CPU (1ms 周期)
         CANopen 协议栈处理
0.8ms ── proto_task 完成
         main_task 获得 CPU (10ms 周期)
         摇杆采集 + 信号链 + 协议发送
2ms ──── safety_task 再次抢占 main_task
         确保安全监控不被阻塞
```

关键设计规则：
- `safety_task` 中**禁止使用 mutex**（避免优先级反转导致安全超时）
- 安全任务只读取数据，使用 atomic 或 volatile 访问
- `protocol_status` 改为 atomic 单写者（proto_task 写，safety_task 无锁读）
- CAN RX 中断处理时间 < 5μs（只做 `osMessageQueuePut` 入队列）
- 协议栈共享状态不再靠 mutex 保护，而是通过 `proto_tx_queue` 收敛为单线程访问
- 所有周期任务使用 `osDelayUntil()` 绝对延时（避免周期漂移）

### 11.9 中断优先级矩阵

v5 将输入采样、协议接收和安全关断的中断优先级显式冻结，避免移植时 `NVIC` 配置漂移：

| 中断源 | 抢占优先级 | 子优先级 | 允许调用 | 禁止事项 | 说明 |
|--------|------------|----------|----------|----------|------|
| `EXTI/SAFE_SHUTDOWN` | 0 | 0 | 直接拉低 STO / 置故障锁存 | 禁止 RTOS API | 最高优先级，确保关断路径不被延迟 |
| `WWDG/EWI` | 1 | 0 | 记录看门狗预警、触发故障锁存 | 禁止阻塞 | 仅做失效前最后记录 |
| `CAN_RX` | 2 | 0 | 读硬件邮箱并入 `can_port_t` 队列 | 禁止协议解析 | ISR 只搬运帧，不碰协议状态 |
| `ADC_DMA_TC` | 3 | 0 | 发布采样完成序号/通知 | 禁止信号处理 | 只发布新快照 |
| `SPI_DMA_TC` | 3 | 1 | 发布传感器新帧 | 禁止值转换 | 与 ADC 同级，子优先级更低 |
| `SysTick` | 7 | 0 | RTOS Tick | 禁止业务逻辑 | 保持在实时外设 ISR 之后 |

约束：
- `CAN_RX` 和 `ADC_DMA_TC` 必须高于 `SysTick`，否则 1ms 周期任务会反向阻塞硬件搬运路径。
- 所有可调用 CMSIS-RTOS API 的 ISR 优先级必须满足 `configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY` 约束。
- 安全关断相关 ISR 不进入 `event_bus`，不依赖任何队列和锁。

## 十二、功能安全架构（v4 新增：E-Gas 三层模型，SIL 2 / PL d）

### 12.1 安全需求与标准对照

工业摇杆用于工程机械、农业设备等安全关键场景，需满足：

| 标准 | 等级 | 核心要求 |
|------|------|---------|
| IEC 61508 | SIL 2 | DC ≥ 90%，安全功能与控制功能独立，程序流监控 |
| ISO 13849 | PL d, Cat.3 | 双通道结构，单一故障不丧失安全功能 |

### 12.2 为什么 v3 不满足

v3 的 `safety_task` 是"旁观者"——高优先级执行，但与 `main_task` 共享 RTOS、内存空间、ADC 驱动、事件总线。**优先级隔离 ≠ 独立性**。RTOS 内核 bug、堆栈溢出、系统节拍丢失都是共因失效点。

### 12.3 E-Gas 三层架构适配

### 12.4 Level 2 — 独立安全计算通道

**核心原则：代码独立、数据独立、执行独立。**

Level 2 的信号计算**必须独立编写**，不能复用 Level 1 的 `axis_pipeline` 代码。IEC 61508-3 7.4.2.2 要求软件多样性——即使算法相同，也应独立实现以避免系统性故障。

#### 12.4.1 L1/L2 硬件独立性要求（v5 新增）

L1/L2 若共用同一 ADC 实例和参考电压，Vref 漂移将导致两路同错——这是 **共因失效 (CCF)**，直接破坏 E-Gas 双通道独立性假设。

**硬件隔离等级**（按推荐程度排列）：

| 等级 | 方案 | 独立性 | 适用场景 |
|------|------|--------|---------|
| 最优 | L2 使用独立 ADC 实例（如 L1→ADC1, L2→ADC2） | 高 | STM32G4 等多 ADC 芯片 |
| 次优 | 同 MCU 不同 ADC + 独立采样触发 + 独立 Vref 监测 | 中 | 双 ADC 但共享部分电源 |
| 最低 | 同 ADC 不同采样序列（必须声明 residual risk） | 低 | 单 ADC 芯片（如 STM32F103），需外部补偿 |

```c
// config/product_descriptor.h — v5 新增：L2 硬件隔离描述
typedef struct {
    uint8_t  l2_adc_instance;              // L2 使用的 ADC 实例编号（如 ADC2）
    uint8_t  l1_adc_instance;              // L1 使用的 ADC 实例编号（如 ADC1）
    uint8_t  l2_adc_channels[MAX_AXES];    // L2 独立 ADC 通道
    uint8_t  l2_vref_monitor_channel;      // Vref 监控通道（VREFINT）
    uint8_t  has_independent_vref;         // 是否具备独立 Vref 监控能力
    uint16_t vref_tolerance_mv;            // Vref 容限（典型 ±30mV）
} safety_channel_hw_profile_t;
```

#### 12.4.3 L2 通道实现（含 VREFINT 监测）

```c
// safety/safety_channel.c — 独立实现，不调用 signal chain 的任何函数
#include "safety_channel.h"

void safety_channel_compute(safety_channel_ctx_t *ctx,
                             safety_channel_output_t *out) {
    out->axes_count = ctx->axes_count;
    out->sensor_ok = 1;

    // v5 新增：先检查 L2 ADC 的 Vref 是否在容限内
    uint32_t vref_mv = 0;
    ctx->l2_adc->read_mv(ctx->vref_channel, &vref_mv);
    if (vref_mv < VREF_NOMINAL_MV - ctx->vref_tol ||
        vref_mv > VREF_NOMINAL_MV + ctx->vref_tol) {
        out->sensor_ok = 0;
        return;  // Vref 异常，L2 数据不可信
    }

    for (uint8_t i = 0; i < ctx->axes_count; i++) {
        uint32_t mv = 0;
        drv_status_t ret = ctx->l2_adc->read_mv(ctx->channel_indices[i], &mv);

        // 独立的开路/短路检测（不复用 fault_detector.c）
        if (ret != DRV_OK || mv < SAFETY_MV_LOW || mv > SAFETY_MV_HIGH) {
            out->sensor_ok = 0;
            out->axes[i] = 0;
            continue;
        }

        // 独立的简化校准（不复用 calibrator.c）
        // 不需要滤波/死区——仅用于比对，精度要求低于 Level 1
        uint16_t raw_approx = (uint16_t)(mv * 4095u / 3300u);
        out->axes[i] = (int32_t)raw_approx - (int32_t)ctx->center[i];
    }
}
```

**硬件层面**：CubeMX 配置中 L1 配 ADC1、L2 配 ADC2，各自独立参考电压通道。若芯片只有一个 ADC（如 STM32F103），则 L2 必须走外部 SPI ADC（如 MCP3208），并在 `safety_channel_hw_profile_t` 中声明 residual risk。

### 12.5 Level 2 — 比较器

```c
// safety/safety_comparator.h
#pragma once
#include <stdint.h>
#include "safety_channel.h"

typedef enum {
    COMPARE_OK,
    COMPARE_DEGRADED,     // 超差但未超阈值
    COMPARE_FAIL,         // 持续超差 → 安全停止
} compare_result_t;

typedef struct {
    int32_t  tolerance[MAX_AXES];    // 允许的 L1-L2 偏差（每轴独立）
    uint8_t  fault_count[MAX_AXES];  // 连续超差计数
    uint8_t  fault_threshold;        // 允许的连续超差次数
} safety_comparator_t;

compare_result_t safety_compare(
    safety_comparator_t *cmp,
    const int32_t *l1_axes,
    const safety_channel_output_t *l2,
    uint8_t axes_count);
```

```c
// safety/safety_comparator.c
#include "safety_comparator.h"

compare_result_t safety_compare(
    safety_comparator_t *cmp,
    const int32_t *l1_axes,
    const safety_channel_output_t *l2,
    uint8_t axes_count)
{
    if (!l2->sensor_ok) {
        return COMPARE_FAIL;
    }

    compare_result_t worst = COMPARE_OK;

    for (uint8_t i = 0; i < axes_count; i++) {
        int32_t diff = l1_axes[i] - l2->axes[i];
        if (diff < 0) { diff = -diff; }

        if (diff > cmp->tolerance[i]) {
            cmp->fault_count[i]++;
            if (cmp->fault_count[i] >= cmp->fault_threshold) {
                worst = COMPARE_FAIL;
            } else if (worst < COMPARE_DEGRADED) {
                worst = COMPARE_DEGRADED;
            }
        } else {
            cmp->fault_count[i] = 0;
        }
    }
    return worst;
}
```

### 12.6 Level 3 — 条件喂狗 + 程序流监控

v3 的 `watchdog_feed()` 是**无条件喂狗**——只要 safety_task 在运行就喂。这不满足 SIL 2 要求。v4 改为条件喂狗：

```c
// safety/safety_watchdog.h
#pragma once
#include <stdint.h>
#include <stdbool.h>

// alive_token: 仅当 Level 2 比较通过时才更新
// 喂狗条件：token 正确 + 程序流标记完整
typedef struct {
    uint32_t expected_token;
    uint32_t actual_token;
    uint32_t cf_expected;       // 程序流期望签名
    uint32_t cf_actual;         // 程序流实际签名
} safety_wdg_ctx_t;

void safety_wdg_init(safety_wdg_ctx_t *ctx);
void safety_wdg_update_token(safety_wdg_ctx_t *ctx);   // L2 比较通过时调用
void safety_wdg_service(safety_wdg_ctx_t *ctx);         // 条件喂狗
```

```c
// safety/safety_watchdog.c
#include "safety_watchdog.h"

// 简单伪随机序列，防止代码意外跳转到 update_token 就通过
static uint32_t next_token(uint32_t seed) {
    return seed * 1103515245u + 12345u;
}

void safety_wdg_init(safety_wdg_ctx_t *ctx) {
    ctx->expected_token = 0xA5A5A5A5u;
    ctx->actual_token   = 0;
    ctx->cf_expected    = CF_MAIN_EXPECTED_SIGNATURE;
    ctx->cf_actual      = CF_SEED;
}

void safety_wdg_update_token(safety_wdg_ctx_t *ctx) {
    ctx->actual_token = ctx->expected_token;
    ctx->expected_token = next_token(ctx->expected_token);
}

void safety_wdg_service(safety_wdg_ctx_t *ctx) {
    bool token_ok = (ctx->actual_token != 0) &&
                    (next_token(ctx->actual_token) == ctx->expected_token);
    bool cf_ok    = (ctx->cf_actual == ctx->cf_expected);

    if (token_ok && cf_ok) {
        IWDG_feed();            // 内部看门狗
        ext_wdg_feed();         // 外部看门狗（如 MAX6369）
    }
    // else: 不喂狗 → 看门狗超时 → 硬件复位

    // 重置程序流标记，等待下一轮
    ctx->cf_actual = CF_SEED;
}
```

### 12.7 Level 3 — 独立硬件关断路径

**最后防线——不经过 RTOS、event_bus、状态机、驱动抽象层。**

v5 改进：将 `safe_shutdown_activate()` 改为平台抽象注入模式，避免安全通用头文件直接 `#include "stm32g4xx.h"` 破坏平台无关边界。实现仍允许直写寄存器，但平台差异被限制在平台目录内。

```c
// safety/safe_shutdown.h — v5: 平台抽象注入（不再包含平台头文件）
#pragma once
#include <stdint.h>

// 平台提供的硬件关断操作
typedef struct {
    void (*activate)(void);     // 硬件直接关断实现（直写 GPIO 寄存器）
} safe_shutdown_ops_t;

// 由 platform_init() 在启动早期注册
void safe_shutdown_register(const safe_shutdown_ops_t *ops);

// 安全关断入口 — 内部调用注册的 ops->activate()
void safe_shutdown_activate(void);
```

```c
// safety/safe_shutdown.c
#include "safe_shutdown.h"

static const safe_shutdown_ops_t *g_ops;
static volatile uint32_t g_safety_shutdown_active;

void safe_shutdown_register(const safe_shutdown_ops_t *ops) {
    g_ops = ops;
}

void safe_shutdown_activate(void) {
    if (g_ops && g_ops->activate) {
        g_ops->activate();
    }
    __atomic_store_n(&g_safety_shutdown_active, 1, __ATOMIC_RELEASE);
    // 尽力通过 CAN 发送紧急帧（不阻塞，不等待）
    // 即使 CAN 发送失败，硬件关断已经生效
}
```

```c
// platform/stm32g4/shutdown_stm32g4.c — STM32G4 实现
#include "stm32g4xx.h"
#include "safe_shutdown.h"

static void stm32g4_shutdown(void) {
    // 直接操作 GPIO 寄存器，不通过 gpio_writer_t 抽象层
    GPIOB->BRR = GPIO_PIN_ENABLE_OUT_Msk;  // 拉低使能输出
}

const safe_shutdown_ops_t shutdown_stm32g4 = { .activate = stm32g4_shutdown };
// 在 platform_init() 中调用: safe_shutdown_register(&shutdown_stm32g4);
```

```c
// platform/s32k144/shutdown_s32k144.c — S32K144 实现
#include "S32K144.h"
#include "safe_shutdown.h"

static void s32k144_shutdown(void) {
    PTB->PCOR = (1u << ENABLE_OUT_PIN);  // 拉低使能输出
}

const safe_shutdown_ops_t shutdown_s32k144 = { .activate = s32k144_shutdown };
```

> **权衡**：增加一层间接调用（函数指针），理论上增加了关断路径的故障点。但函数指针存储在 `.rodata`（Flash），被 CRC 校验覆盖，风险可接受。若认证审计不接受，可改为 `#if defined(PLATFORM_STM32G4)` 条件编译——仅此一处破坏平台无关性。

### 12.10 附加诊断 — 平台认证安全库集成（DC ≥ 90%）

底层硬件诊断**不自研**，使用平台厂商提供的已认证安全库：

| 平台 | 安全库 | 认证等级 | 覆盖项 |
|------|--------|---------|--------|
| STM32G4 | **X-CUBE-STL** (ST 官方) | IEC 61508 SIL 2/3 | CPU/RAM/Flash/Clock 自检 |
| S32K144 | **SafeAssure Safety SW** (NXP) | IEC 61508 SIL 2, ISO 26262 ASIL B | CST(CPU)/ECC(RAM)/CRC(Flash)/CMU(Clock) |

> 使用预认证库的优势：减少 80%+ 的认证文档工作量，厂商提供 FMEDA 和安全手册，审核时直接引用。

#### 12.10.4 safety_task 中调用（v5: 分片化 + 供电电压监控）

v5 将诊断改为预算化分片执行，每个 safety_task 周期只执行一个诊断步骤，确保 WCET 可控：

```c
// safety/safety_diagnostics.h — v5: 分片诊断步骤枚举
typedef enum {
    SAFETY_DIAG_CPU_REG,
    SAFETY_DIAG_RAM_SLICE,
    SAFETY_DIAG_FLASH_CRC_SLICE,
    SAFETY_DIAG_CLOCK,
    SAFETY_DIAG_SUPPLY_VOLTAGE,    // v5 新增：供电电压监控
    SAFETY_DIAG_STEP_COUNT
} safety_diag_step_t;

// 执行一个诊断步骤，budget_us 为本次允许的最大执行时间
bool safety_diagnostics_step(safety_diag_step_t step, uint32_t budget_us);
```

```c
// safety_task 中的附加诊断调用
static safety_diag_step_t g_diag_step = 0;

static bool safety_diagnostics_step(safety_diag_step_t step, uint32_t budget_us) {
    const safety_stl_t *stl = platform_get_stl();
    (void)budget_us;  // 未来可用于动态预算控制

    switch (step % SAFETY_DIAG_STEP_COUNT) {
        case SAFETY_DIAG_CPU_REG:
            if (stl->cpu_test_partial() == STL_RESULT_FAIL) {
                safe_shutdown_activate();
                return false;
            }
            break;

        case SAFETY_DIAG_RAM_SLICE:
            if (stl->ram_test_partial() == STL_RESULT_FAIL) {
                safe_shutdown_activate();
                return false;
            }
            break;

        case SAFETY_DIAG_CLOCK:
            if (stl->clock_cross_check() == STL_RESULT_FAIL) {
                safe_shutdown_activate();
                return false;
            }
            break;

        case SAFETY_DIAG_FLASH_CRC_SLICE: {
            // Flash CRC 分片执行，每 SAFETY_DIAG_STEP_COUNT 轮检查一片
            static uint16_t flash_round = 0;
            if (++flash_round >= 100) {  // 约每 1s 完成一轮全覆盖
                flash_round = 0;
                if (stl->flash_crc_check() == STL_RESULT_FAIL) {
                    safe_shutdown_activate();
                    return false;
                }
            }
            break;
        }

        case SAFETY_DIAG_SUPPLY_VOLTAGE: {
            // v5 新增：供电电压监控
            // 传感器供电异常（如 5V 跌落到 4V）会导致 L1/L2 系统性偏移
            uint32_t supply_mv = 0;
            safety_channel_ctx_t *l2_ch = app_get_l2_channel();
            l2_ch->l2_adc->read_mv(ADC_CH_SUPPLY, &supply_mv);
            if (supply_mv < SUPPLY_MIN_MV || supply_mv > SUPPLY_MAX_MV) {
                event_bus_post(&(event_t){
                    .type = EVT_SAFETY_STATE_CHANGED,
                    .data.safety_state = 2 }, EVT_PRIO_SAFETY);
            }
            break;
        }
    }
    return true;
}
```

> **供电电压阈值**：`SUPPLY_MIN_MV` / `SUPPLY_MAX_MV` 由产品配置定义（典型 5V±10%: 4500mV ~ 5500mV）。`ADC_CH_SUPPLY` 已在 `adc_channel_id_t` 中定义但此前未使用。

#### 12.10.5 安全变量互补存储（应用层，仍需自研）

认证库不覆盖应用层的安全关键变量保护，此部分仍需自研：

```c
// safety/safe_var.h — 安全关键变量互补存储
#pragma once
#include <stdint.h>
#include <stdbool.h>

typedef struct {
    int32_t  value;
    int32_t  complement;   // 应等于 ~value
} safe_var_t;

static inline bool safe_var_check(const safe_var_t *v) {
    return v->value == (int32_t)(~(uint32_t)v->complement);
}

static inline void safe_var_write(safe_var_t *v, int32_t val) {
    v->value      = val;
    v->complement = (int32_t)(~(uint32_t)val);
}
```

### 12.13 上电自检 POST（v5 新增）

IEC 61508 要求上电前验证硬件完好。v4 只有运行时周期自检，缺少启动阶段的安全闸门。v5 将 POST 前置为状态机入口，在 RTOS 启动前完成。

**启动顺序**：`HAL_Init` → `platform_init` → `safe_shutdown_register()` → **`safety_post_run()`** → POST 通过后 `osKernelStart`

```c
// safety/safety_post.h — 上电全量自检
#pragma once
#include "safety_stl.h"
#include <stdbool.h>

typedef enum {
    POST_STAGE_CPU,
    POST_STAGE_RAM,
    POST_STAGE_FLASH,
    POST_STAGE_CLOCK,
    POST_STAGE_STACK_INIT,
    POST_STAGE_ADC_SELF_TEST,
    POST_STAGE_WDG_VERIFY,
    POST_STAGE_RELAY_TEST,
    POST_STAGE_COMPLETE
} post_stage_t;

typedef struct {
    post_stage_t current_stage;
    uint32_t     fail_mask;       // 位掩码记录失败项
} post_ctx_t;

// 返回 true = 全部通过，false = 有失败项
bool safety_post_run(post_ctx_t *ctx, const safety_stl_t *stl);
```

```c
// safety/safety_post.c
#include "safety_post.h"
#include "safe_shutdown.h"

bool safety_post_run(post_ctx_t *ctx, const safety_stl_t *stl) {
    ctx->fail_mask = 0;

    // CPU 全量自检（POST 阶段不分片，一次跑完）
    if (stl->cpu_test_partial() == STL_RESULT_FAIL) {
        ctx->fail_mask |= (1u << POST_STAGE_CPU);
    }

    // RAM March C-（POST 阶段跑全量，非分片）
    if (stl->ram_test_partial() == STL_RESULT_FAIL) {
        ctx->fail_mask |= (1u << POST_STAGE_RAM);
    }

    // Flash CRC 全量校验
    if (stl->flash_crc_check() == STL_RESULT_FAIL) {
        ctx->fail_mask |= (1u << POST_STAGE_FLASH);
    }

    // 时钟交叉校验
    if (stl->clock_cross_check() == STL_RESULT_FAIL) {
        ctx->fail_mask |= (1u << POST_STAGE_CLOCK);
    }

    // MPU / Stack Sentinel 初始化
    stl->stack_overflow_init();

    // ADC 自检：读已知参考通道（内部 Vrefint），校验是否在容限内
    // WDG 验证：确认 WDG 能正确超时
    // 安全继电器测试：短暂激活-释放，确认继电器响应

    return ctx->fail_mask == 0;
}
```

```c
// app/device_state.c — STATE_BOOT 集成 POST
static drv_status_t boot_self_test(void) {
    static post_ctx_t post_ctx;
    const safety_stl_t *stl = platform_get_stl();

    if (stl == NULL) {
        // 非安全型号，跳过 POST
        return DRV_OK;
    }

    if (!safety_post_run(&post_ctx, stl)) {
        // POST 失败，不允许进入 PRE_OPERATIONAL
        nv_storage_write_post_result(post_ctx.fail_mask);
        safe_shutdown_activate();
        return DRV_ERROR;
    }
    return DRV_OK;
}
```

> **关键**：`safety_stl_t` 接口需区分 POST 模式（全量、允许阻塞几百 ms）和运行时模式（分片、每次 < 300μs）。建议增加 `cpu_test_full()` / `ram_test_full()` 接口或传入 mode 参数。

## 十五点五、资源预算与实现余量

资源预算按 `JS-300 + CANopen + SAFETY_ENABLED` 基线配置冻结，用于实现阶段约束 WCET、栈和存储上界。

### 15.5.1 任务级 WCET / 栈预算

| 执行单元 | 周期 | WCET 预算 | 栈预算 | 说明 |
|----------|------|-----------|--------|------|
| `safety_task` | 2 ms | 0.60 ms | 1536 B | L2 采集、比较、条件喂狗、单步诊断 |
| `proto_task` | 1 ms | 0.35 ms | 1536 B | `proto_tx_queue` 出队、`on_rx_frame`、`process_periodic` |
| `ctrl_task` | 5 ms | 0.25 ms | 1024 B | 状态机推进、事件分发 |
| `main_task` | 10 ms | 1.20 ms | 2048 B | 采样快照、信号链、`output_adapt`、投递 TX |
| `diag_task` | 100 ms | 2.00 ms | 1024 B | 日志、NV flush、健康检查 |
| `CAN_RX ISR` | 事件驱动 | 5 us | 96 B | 只搬运一帧到队列 |
| `ADC_DMA_TC ISR` | 事件驱动 | 4 us | 96 B | 只发布 seq 和通知 |

### 15.5.2 Flash / RAM 预算

| 模块 | Flash 预算 | RAM 预算 | 说明 |
|------|------------|----------|------|
| 平台驱动 + HAL 适配 | 48 KB | 6 KB | ADC/CAN/GPIO/SPI/STL 适配 |
| 输入采集 + 信号链 | 18 KB | 3 KB | `input_acq`、`scan_buffer_t`、pipeline |
| 协议栈 | 28 KB | 4 KB | CANopen/J1939 引擎和队列 |
| 安全架构 | 24 KB | 4 KB | L2/L3、POST、诊断分片 |
| 事件/服务/应用组装 | 14 KB | 3 KB | event bus、NV、logging、app |
| 裕量 | 28 KB | 12 KB | 用于新 profile、诊断扩展、认证插桩 |
| 合计预算 | 160 KB | 32 KB | 基于单 MCU 中档配置冻结 |

### 15.5.3 预算约束

- 任一任务实测 WCET 超过表内预算的 80% 时，必须更新预算表并重新评估调度裕量。
- `proto_task` 的协议内部状态只允许单线程访问；若引入第二发送路径，预算立即失效。
- `scan_buffer_t`、`proto_tx_queue`、事件队列的 RAM 占用必须计入产品变体差异表，不允许隐式增长。

## 十六、设计原则对照总结

### SOLID 原则合规性

| 原则 | 原设计状态 | 改进方案 | 改进后状态 |
|------|-----------|---------|-----------|
| **S** — 单一职责 | `joystick_core.c` 五职责合一；`main_app.c` 混合组装与运行；协议引擎兼做值转换+打包 | 信号管线拆分；组装与运行分离；v4 新增 output_adapter 将值转换从协议引擎剥离 | 每个模块只有一个变更理由 |
| **O** — 开闭原则 | `product_config.h` 的 `#elif` 链；CMake 中硬编码安全型号名 | 每产品独立 `.c`；v4 改为 `SAFETY_ENABLED` 参数驱动 | 新增产品/安全型号不修改任何已有代码 |
| **L** — 里氏替换 | 协议接口强制所有方法，部分协议需写空函数 | 核心接口 + 能力位掩码 + 可选扩展方法 | 所有 CAN 协议都可以安全替换使用 |
| **I** — 接口隔离 | `adc_driver_t` 包含所有方法 | 拆分为 `adc_reader` / `adc_voltage_reader` / `adc_continuous` | 每个模块只依赖自己需要的最小接口 |
| **D** — 依赖倒置 | `#if defined(PLATFORM_STM32G4)` 硬编码平台 | 平台注册表 + 显式依赖注入 | 上层完全不知道下层具体实现 |

### 设计模式使用

| 模式 | 用途 | 位置 |
|------|------|------|
| **观察者模式** | 模块间松耦合事件通信 | `event/event_bus.h` |
| **策略模式** | 可插拔的死区/校准算法 | `deadzone_fn_t` / `calibrate_fn_t` |
| **状态模式** | 设备生命周期管理 | `app/device_state.h` |
| **扫描-缓冲模式** | 硬件采集与数据消费解耦 | `input/scan_buffer.h` + `input/input_acq.h` |
| **Adapter 模式** | v4: 管线内部值 → 自描述输出 | `output/output_adapter.h` |
| **Anti-Corruption Layer** | v4: 协议层不解释管线内部表示 | `output_data_t` 自描述边界对象 |
| **Tagged Union** | v4: 编译期类型安全的协议配置 | `protocol_bundle_t` |
| **双通道比较** | v4: E-Gas L2 独立计算+比对 | `safety/safety_comparator.h` |

### v4 相比 v3 的关键改进

| 改进点 | v3 设计 | v4 改进 | 原因 |
|--------|---------|---------|------|
| 信号链接口 | 泛型 chain（`int16_t`）+ 显式 pipeline（`int32_t`）并存 | 删除泛型 chain，统一为显式 `axis_pipeline`（`uint16_t→int32_t` 分界） | 消除类型不匹配和概念混淆 |
| 值域转换 | 嵌入协议引擎内部 | 独立 `output_adapter` 层（纯函数） | SRP + 可复用（HMI/日志/调试均可调用） |
| 协议引擎输入 | `joystick_data_t`（非自描述） | `output_data_t`（自描述，含值范围） | Anti-Corruption Layer |
| max_deflection | 单一值（假设所有轴同分辨率） | per-axis（`uint16_t max_deflection[MAX_AXES]`） | 支持 JS-300 混合 12-bit/14-bit 传感器 |
| 产品描述符 | 缺少 protocol 字段 | 新增 `protocol_bundle` + `output_maps` + `safety_config` | 完整的产品自描述 |
| 安全架构 | safety_task 旁观式监控（读 L1 数据判断异常） | E-Gas 三层：L2 独立采集+比对，L3 条件喂狗+硬件关断 | SIL 2 / PL d 合规 |
| 看门狗 | 无条件喂狗 | 条件喂狗（L2 token + 程序流标记双重校验） | IEC 61508-7 要求 |
| 安全关断 | 依赖 event_bus → 状态机（共因失效） | GPIO 直驱安全继电器（不经 RTOS/event_bus） | 独立关断路径 |
| CMake 安全模块 | 硬编码 `JS300 OR JS400` | `SAFETY_ENABLED` 参数驱动 | OCP：新增安全型号不改 CMakeLists.txt |

### 关键约束

所有改进都保持了嵌入式 C 的实用性与功能安全合规性：
- 纯 C99 标准，无 GCC/编译器特有扩展（`_Atomic` 来自 C11，安全模块需指定 `-std=c11`）
- 无虚函数表 / RTTI
- 无动态内存分配（`malloc`/`free`）
- 无 C++ 异常
- 所有缓冲区静态分配
- 事件总线订阅者数量编译时确定
- 所有初始化顺序显式可控（无 constructor / 隐式初始化）
- MISRA C:2012 可达标
- IEC 61508 SIL 2 / ISO 13849 PL d 架构合规（E-Gas 三层 + 独立关断路径）
